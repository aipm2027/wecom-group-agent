## 3. 评测发现的真实问题台账(每条都有修复闭环)

> 这是评测体系的价值证明:以下问题**没有一个能被单元测试发现**——单测全绿时它们全部存在。

### 记录 #1(2026-07-10 上午)· 断言假设错误:RAG top-k 凑数

- **现象**:离线评测首跑 15/17,`rag-shipping`/`rag-invoice` 的 must_not 断言失败(税号/顺丰混入召回)。
- **定性**:**断言错,代码"没错"**——`_search` 语义 top-k 无分数阈值,零分 chunk 凑满坑位是设计现状。
- **处置**:撤掉两条 must_not + 把"top-k 无阈值"记入路线图 → P2-7c 落地相对阈值后**断言加回并通过**。完整闭环 3 轮。

### 记录 #2(2026-07-10 上午)· 互审抓 bug:evals 引错函数名

- **现象**:review 对面 admin_console 时发现它 `from main import load_env_file` 与我 evals 里的 `_load_dotenv` 不一致。
- **定性**:**我的 bug**——`--online` 的 .env 加载引错名字,且被 try/except 吞掉会静默跳过。
- **处置**:`1ef6ab0` 修复。互审机制首日即双向抓真问题。

### 记录 #3(2026-07-10 下午)· persona 误召 ×2(在线评测抓出)

- **现象**:在线首轮 26/27:"已经拆封了还能退吗"被转人工(reason=AI 判定复杂场景);修完后"再便宜点,不然我不买了"连续 2 跑被转。
- **定性**:**行为错**——persona 把"政策咨询"与"实际诉求"混为一谈("退换、退款…转人工"一刀切);砍价条款太宽。
- **处置**:persona 售后节+触发条款 #2/#3 改写(咨询按知识作答;普通砍价先标准话术,坚持才转);案例本身也修正(带流失威胁的坚持砍价按 persona 可转,不予断言)。修后转人工类目 7/7 双跑稳定。

### 记录 #4(2026-07-10 下午)· 规则层回归:政策名词误伤(对面 #13 review 抓出)

- **现象**:对面用 5 个探针实测:"怎么退货呀/退货流程是什么/退款一般几天到账/支持七天无理由退货吗"全被规则层硬触发——**规则层把 persona 刚修好的误召又确定性引回来了**。
- **定性**:关键词集"宽在名词、漏在句式"——"退款/退货"是政策名词,命中≠诉求;硬触发会覆盖 LLM 按 persona 做出的正确判断。
- **处置**:`de3c8f2` 售后关键词拆两档(损害词硬触发/政策名词须配诉求标记);5 探针进金标+单测。对面复验 9 探针 8 中通过("给我退了吧"落 LLM 兜底,属设计接受)。

### 记录 #5(2026-07-10 下午)· 知识缺口:"退款几天到账"(在线评测抓出)

- **现象**:规则层修干净后,该探针在线仍失败——真 LLM 把它转了人工。
- **定性**:**LLM 是对的**——样例知识库没有退款时效条目,persona"知识答不了→转人工"被正确执行;缺口在知识内容。
- **处置**:`prompts/knowledge.md` 补"退款到账 1–3 工作日"条目,双跑稳定通过。⇒ 沉淀出"误召三层联动"认知(见 [09-A2](../技术文档/09-关键难点与决策记录.md))。

### 记录 #6(2026-07-10 下午)· 断言误伤:否定句(门禁首跑抓出)

- **现象**:`make gate` 首跑 16/17:合规回复"真没法打包票**保证明天**必到"被 `must_not_include: ["保证明天"]` 误杀。
- **定性**:**断言错**——子串匹配天生不懂否定语义。
- **处置**:该方向断言改由 LLM-as-judge 语义判定,子串只验"出现拒绝措辞";修后门禁 17/17 放行。

### 记录 #7(2026-07-10)· 安全同款三连:Content-Length 负值

- **来源**:admin_console 三镜头审查(20 subagent,12 真/5 假阳性)发现第一处 → 全库 grep 出 api_server 同款 → 顺藤摸瓜查出 **wecom_kf 公网回调口**同款(严重度最高:验签前可触发线程阻塞)。
- **处置**:三处统一 `max(0, int(...))+except ValueError` 修法(`469544b`/`8f6c488`),kf 侧补 `_parse_content_length` 单测 5 断言。

### 记录 #8(2026-07-10)· 智能机器人端到端验收 + 长跑稳定性(真机生产环境)

- **现象**:真实用户(FengXianYu)企微单聊机器人「咕咕嘎嘎」,**12 条完整往返**(sessions.db `aibot-single-FengXianYu` 会话可查,由 CONV-B 独立还原核验)。链路层全对:订阅/回调/去重/回复/SQLite 持久化零问题;"你有什么卖的→看看"多轮商品上下文正确。
- **长跑数据**:当班实例 31+ 分钟、前一实例 ~50 分钟,均**单次订阅零意外重连**(日志仅 1 条"订阅成功",无重连退避记录),期间承载上述真实对话;一次计划内重启(nit 修复版上线)重连即恢复。
- **定性**:链路=通过;回复质量=4 项细节问题(连续三条"hi亲~在的!"复读开场/"毛线"被曲解为"没事"/`**markdown**` 星号原样透出微信/每条 😊)——**全部属 persona 层,非代码缺陷**。
- **处置**:persona 说话风格+输出格式节迭代 + evals 多轮(turns)支持 + 3 个细节回归案例(CONV-C,见 DISCUSSION #25[C 楼]);修后 `make gate` 全绿方可上线。~~待补:双实例互踢行为实测~~ → **已于同日实测**(非计划,见 DISCUSSION #36[C 楼]):双 agent 并存触发 ≤30s 周期互踢风暴,烧掉订阅限频(45009),指数退避 8s→16s→30s 后**自愈重连成功**——退避设计与「调试前停生产」警告双双得到真实验证(处置:留带 outbox 桥的新进程,kill 旧进程)。

### 记录 #9(2026-07-10)· 弱 AI 维护菜谱实操走查(第三轮整理:实操验证层)

- **方法**:全新 `git clone` 到沙盒(无 .env、无本机状态),扮演"只会照 CLAUDE.md 菜谱操作的弱 AI",逐条真实执行菜谱并观察卡点。
- **结果(全部通过)**:① 干净克隆 `make check` **开箱即绿**(离线体系零本机依赖);② 菜谱「加商品」:只改 products.json(故意不动 knowledge.md)→ 评测/护栏绿、新品立即可精确查价——护栏只锁"两边都提到的商品价格一致",不强迫每个 SKU 进知识库,口径合理;③ 菜谱「改售后政策」:knowledge.md 小改 → 全绿;④ 菜谱「加转人工关键词」:`_ESCALATION_RULES` 加"霸王条款"→ 探针即时命中、takeover 套件绿;⑤ console 界面文案改动:测试不锁文案(改中文/CSS 安全)。
- **发现的真卡点(已修)**:文档/注释里**硬编码测试套数反复过期**(10→12→13→15→16 一路人肉追改,本轮又抓到 2 处 15)——处置:高频处一律改为"全部离线测试(以 `make test` 输出为准)"去计数化;`main.py`/`admin_console.py` 头部补「维护者注意」(加适配器三步走/console 哪能随便改哪要跑测试)。
- **结论**:三层维护保障(手册/护栏/实操验证)闭环,弱 AI 按菜谱操作全程无卡点。

审查类工作流均带"逐条对抗验证"环节,以下发现曾被提出并被**行号级证据驳回**,后续不必重查:

- admin_console XSS(onclick 实体转义):技术上成立,但 chat_id 链路攻击者不可控(企微系统生成)+口令后台,判非漏洞(远期建议 addEventListener);
- SSRF/路径穿越(代理层):回环绑定+强鉴权+api_server 路由不解析 `..`,利用链断裂;
- `_sessions` 只增不减:logout 有 discard,单租户内存量级可忽略;
- login/logout 缺 X-Requested-With:文档限定"代理的写操作",login 是鉴权边界本身,CSRF 无额外收益;
- 更早一轮的 59 条全局审查、24 条路线图审计驳回清单分别见 [TASKS.md](../../TASKS.md) 与 [产品化路线图附录](../产品化路线图.md)。
