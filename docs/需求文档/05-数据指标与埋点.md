# 需求文档 · 数据指标与埋点

> 本文件定义运营/效果指标与埋点方案，与 `/api/metrics` 已实现字段对齐。返回 [需求总览](00-总览.md)

## 1. 北极星指标与核心指标

### 1.1 北极星指标

| 指标 | 定义 | 计算方式 | 现状 |
|------|------|----------|------|
| **人工接管率** | 需要人工介入的会话数 / 总会话数 | `needs_human / conversations` | `/api/metrics` 已输出原始计数，运营后台需自行计算比率。 |
| **回复覆盖率** | agent 成功发出回复的消息数 / 触发回复的消息数 | 需 `MetricEvent` 表落地后统计（见第 3 节） | **规划**（`MetricEvent` 未实现）。 |

### 1.2 核心运营指标

| 指标 | 定义 | 计算方式 | 现状 |
|------|------|----------|------|
| 会话总量 | 当前活跃会话数 | `conversations` | `/api/metrics` 已返回。 |
| 待人工队列长度 | 已触发转人工且未被接管的会话数 | `needs_human`（`take_over()` 会清除 `needs_human`，故该计数仅含"待接管"、不含已接管；精确列表用 `/api/queue`） | `/api/queue` 已返回 `needs_human=True` 且 `human_controlled=False` 的会话列表。 |
| 人工接管中 | 当前处于人工接管状态的会话数 | `human_controlled` | `/api/metrics` 已返回。 |
| 消息总量 | 所有会话的历史消息条数之和 | `messages` | `/api/metrics` 已返回。 |
| LLM 调用次数 | LLMHandler 调用 API 的次数 | 需 `MetricEvent` 表落地后统计 | **规划**。 |
| LLM 降级次数 | 网络异常/超时/解析失败时 fallback 的次数 | 需 `MetricEvent` 表落地后统计 | **规划**。 |
| 平均响应延迟 | 从收到消息到发出回复的耗时 | 需 `MetricEvent` 表落地后统计 | **规划**。 |
| 转人工率 | needs_human 会话数 / 总会话数 | `needs_human / conversations` | 可由 `/api/metrics` 字段计算。 |
| 人工接管率 | human_controlled 会话数 / 总会话数 | `human_controlled / conversations` | 可由 `/api/metrics` 字段计算。 |

## 2. 会话级/客户级埋点字段

### 2.1 已落地：Session 运行时字段

`Session` 对象（[`core/session.py`](../../core/session.py)）已包含以下可直接用于运营指标的状态字段：

| 字段 | 类型 | 说明 | API 暴露 |
|------|------|------|----------|
| `chat_id` | `str` | 会话唯一标识 | `/api/conversations` |
| `human_controlled` | `bool` | 是否处于人工接管态 | `/api/conversations`, `/api/metrics` |
| `needs_human` | `bool` | 是否已触发转人工 | `/api/conversations`, `/api/metrics` |
| `escalation_reason` | `str` | 转人工原因摘要 | `/api/conversations` |
| `history` | `deque[Message]` | 最近 `MAX_CONTEXT=20` 条消息 | `/api/conversations/{id}/messages` |

### 2.2 规划：MetricEvent 表字段

> **状态：规划（P2）。** 当前系统尚未实现事件级持久化表，以下字段为未来 `MetricEvent` 表的设计草案，供运营后台与离线分析使用。

| 字段 | 类型 | 说明 |
|------|------|------|
| `event_id` | `int` | 自增主键 |
| `event_type` | `str` | 事件类型：`reply`（agent 回复）、`escalation`（转人工）、`rate_limit_hit`（限流命中）、`llm_error`（LLM 调用失败）、`human_takeover`（人工接管）、`human_release`（人工释放）、`dedup_hit`（去重命中） |
| `chat_id` | `str` | 关联会话（nullable） |
| `customer_id` | `str` | 关联客户（nullable） |
| `payload` | `dict` | 事件详情（JSON），如 `{"model": "step-3.7-flash", "latency_ms": 1200, "fallback": true}` |
| `timestamp` | `datetime` | 事件时间 |

> **依赖说明**：`MetricEvent` 的落地需要扩展 `SqliteSessionStore` 或新增独立的事件存储层。当前 SQLite 只持久化了 `sessions` 和 `messages` 两张表，未记录事件级明细。

## 3. 指标采集与 API 对齐

### 3.1 已实现的 `/api/metrics`

`GET /api/metrics` 当前返回以下字段（源码：[`api_server.py`](../../api_server.py) 的 `_metrics()` 方法）：

```json
{
  "conversations": 12,
  "needs_human": 2,
  "human_controlled": 1,
  "messages": 156
}
```

| 字段 | 说明 | 数据来源 |
|------|------|----------|
| `conversations` | 当前活跃会话总数（`sessions.all()` 的会话数） | 内存 `SessionStore` 或 `SqliteSessionStore` 的 `all()` |
| `needs_human` | `needs_human=True` 的会话数 | `sum(1 for s in sessions if s.needs_human)` |
| `human_controlled` | `human_controlled=True` 的会话数 | `sum(1 for s in sessions if s.human_controlled)` |
| `messages` | 所有会话历史消息条数之和 | `sum(len(s.history) for s in sessions)` |

### 3.2 `/api/queue`：待接管队列

`GET /api/queue` 返回 `needs_human=True` 且 `human_controlled=False` 的会话列表，是人工工作台“收件箱”的核心数据来源。

### 3.3 规划中的指标端点（P2）

| 端点 | 方法 | 说明 | 状态 |
|------|------|------|------|
| `/api/metrics/dashboard` | `GET` | 今日消息量、回复量、触发率、LLM 错误率、人工接管率等聚合看板 | **规划** |
| `/api/metrics/events` | `GET` | 事件查询，`?type=reply&chat_id=...&from=...&to=...`，分页 | **规划** |
| `/api/metrics/llm_stats` | `GET` | LLM 调用统计：次数、平均延迟、降级次数、token 消耗（估算） | **规划** |
| `/api/metrics/events` | `POST` | 手动上报事件（测试用） | **规划** |

## 4. 报表与复盘口径

### 4.1 日报表（基于当前 `/api/metrics` 可计算）

| 报表项 | 口径 | 当前可行度 |
|--------|------|-----------|
| 总会话数 | `conversations` | 高 |
| 转人工会话数 | `needs_human` | 高 |
| 人工接管会话数 | `human_controlled` | 高 |
| 消息总量 | `messages` | 高 |
| 平均会话消息数 | `messages / conversations` | 高 |
| 待处理队列长度 | `/api/queue` 返回列表长度 | 高 |

### 4.2 周/月报表（依赖 MetricEvent，规划中）

| 报表项 | 口径 | 当前可行度 |
|--------|------|-----------|
| LLM 平均响应延迟 | `avg(latency_ms) where event_type='reply'` | 低（需 MetricEvent） |
| LLM 降级率 | `count(fallback=true) / count(event_type='reply')` | 低（需 MetricEvent） |
| 限流命中率 | `count(event_type='rate_limit_hit') / count(event_type='reply')` | 低（需 MetricEvent） |
| 去重命中率 | `count(event_type='dedup_hit') / count(event_type='reply')` | 低（需 MetricEvent） |
| 转人工原因分布 | `group by escalation_reason` | 低（需 MetricEvent 或扩展 Session 表） |
| 高峰时段分布 | `group by hour(timestamp)` | 低（需 MetricEvent） |

## 5. 埋点实施建议

### 5.1 短期（当前可用）

- **运营后台**：轮询 `GET /api/metrics` 展示实时会话数、转人工数、接管数、消息总量。
- **人工工作台**：调用 `GET /api/queue` 获取待处理会话列表，调用 `POST /api/conversations/{id}/takeover` 接管，`POST /api/conversations/{id}/release` 释放。
- **告警**：基于 `needs_human` 和 `human_controlled` 的差值（待处理队列）设置阈值告警。

### 5.2 中期（P1/P2）

1. **扩展 SQLite Schema**：在 [`core/session_sqlite.py`](../../core/session_sqlite.py) 的 `_SCHEMA` 中新增 `events` 表（或独立事件存储），记录 `MetricEvent` 明细。
2. **Router/Handler 埋点**：在 `Router.on_message` 和 `LLMHandler.reply` 中插入事件记录：
   - 收到消息时记录 `event_type='message_received'`
   - 去重命中时记录 `event_type='dedup_hit'`
   - 限流命中时记录 `event_type='rate_limit_hit'`
   - LLM 调用成功时记录 `event_type='reply'`，附带 `latency_ms`、`model`
   - LLM 调用失败降级时记录 `event_type='llm_error'`，附带 `error_type`
   - 转人工时记录 `event_type='escalation'`，附带 `reason`
   - 人工接管/释放时记录 `event_type='human_takeover'` / `'human_release'`
3. **聚合 API**：基于 `events` 表实现 `/api/metrics/dashboard` 和 `/api/metrics/llm_stats`。

### 5.3 长期（P2 以后）

- **客户画像**：从 `messages` 表聚合 `customer_id` 维度数据，生成客户标签、活跃度、客单价等。
- **离线分析**：定期导出 `events` 表至数据仓库（如 ClickHouse/BigQuery），支持 SQL 级灵活分析。
- **A/B 实验**：基于 `event_type='reply'` 的 `payload.model` 字段，对比不同 LLM 模型的回复质量与人工接管率。
