# 需求文档 · 竞品与市场调研

> 本文件沉淀私域带货客服/SCRM 赛道的竞品与市场调研。返回 [需求总览](00-总览.md)

## 1. 竞品盘点

### 1.1 企微群 Hook 类工具（非合规方案）

| 竞品/方案 | 能力 | 平台 | 合规风险 | 本项目对比 |
|-----------|------|------|----------|-----------|
| ntwork + 企微客户端注入 | 接管真实企微群/单聊，读取群消息并自动回复 | 仅 Windows | **高**：违反《企业微信用户协议》，有封号风险 | 本项目 `NtworkWecomAdapter` 已预留接口（[`adapters/ntwork_wecom.py`](../../adapters/ntwork_wecom.py)），但当前仍为 **stub（仅抛 NotImplementedError）**，待 Windows 实机补齐。架构上已解耦，接入成本极低。 |
| wechaty / itchat 等微信个人号框架 | 个人号微信群机器人，自动回复、群发、加好友 | 跨平台（但个人号更易被封） | **极高**：微信对个人号机器人打击严厉，封号概率大 | 本项目不做个人号，聚焦企业微信 + 官方客服合规通道。 |
| 第三方企微 SCRM（如微伴、尘锋、句子互动） | 提供群发、客户画像、标签、自动回复等完整 SCRM 功能 | SaaS/私有化 | 低：走官方接口 | 本项目当前不覆盖群发、标签画像等完整 SCRM 能力，定位为“轻量级 Agent 回复大脑”，可与 SCRM 互补。 |

### 1.2 官方合规方案

| 方案 | 能力 | 局限 | 本项目状态 |
|------|------|------|-----------|
| **微信客服（企业微信客服）API** | 官方合规 1:1 客服通道，无封号风险；支持回调 + sync_msg 拉取消息 | 只能单聊，**无法进群**；需公网回调地址 + AES 解密 | **已实现**（[`adapters/wecom_kf.py`](../../adapters/wecom_kf.py)）：回调 HTTP 服务 + msg_signature 验签 + AES 解密 + sync_msg 拉取 + send_msg 下发；access_token 带过期缓存；HTTP 客户端可注入离线测试。真机与腾讯服务器互通尚待公网回调联调。 |
| **企业微信自建应用** | 官方开放接口，可接收消息、发送应用消息 | 群消息只能被动接收（通过回调），无法主动读取历史群聊内容；能力有限 | 本项目未采用此路线，因其无法满足“实时群聊回复”需求。 |

### 1.3 LLM 客服 SaaS

| 竞品 | 特点 | 与本项目差异 |
|------|------|-------------|
| 智齿科技 / 网易七鱼 / 环信等 | 成熟 SaaS，提供知识库、工单、人工客服、报表完整闭环 | 本项目为自建方案，零第三方依赖、可私有化部署、LLM 模型可自由切换（当前接 StepFun，未来可换），但需自行维护基础设施。 |
| 基于大模型的智能客服（如部分创业公司方案） | 强调“对话即服务”，快速接入 | 本项目聚焦私域电商场景，深度整合企微生态；且强调“传输无关”，可适配多种消息通道。 |

## 2. 差异化与定位

### 2.1 我们的核心差异点

1. **传输无关三层架构**：Adapter（传输）↔ Router（去重/触发/限流/接管）↔ Handler（回复大脑）。换传输（Mock → 微信客服 → 未来企微 hook）不改核心逻辑。
2. **零第三方依赖，纯 Python 标准库**：核心及所有适配器实现均不依赖 pip 包，部署成本极低。
3. **知识模块可插拔四级阶梯**：`static`（全量塞）→ `rag`（语义检索）→ `structured`（结构化商品库）→ `hybrid`（组合），通过环境变量 `KNOWLEDGE_PROVIDER` 切换，无需改代码。
4. **人工接管闭环**：LLM 输出 `[[转人工]]` 标记 → 剥离后触发 `session.mark_needs_human` → Router 回调 → REST API `/api/queue` + `takeover`/`release` 实现人工工作台基础能力。
5. **会话持久化**：`SqliteSessionStore` 已落地，重启后最近 20 条历史与接管/转人工状态可恢复。

### 2.2 目标市场判断

| 维度 | 判断 |
|------|------|
| **市场规模** | 电商私域客服是刚需，尤其高客单价、复购率高的品类（食品、美妆、母婴）。头部商家已普遍使用 SCRM，但中小商家对“低成本、可私有化、模型可切换”的智能客服有明确需求。 |
| **合规趋势** | 腾讯对非官方 hook 打击力度持续加大，**合规通道（微信客服）是长期趋势**。本项目“合规优先 + 预留 hook 扩展”的策略符合监管方向。 |
| **技术趋势** | LLM 成本持续下降（阶跃星辰、DeepSeek 等国产模型性价比提升），“大模型 + 私域”从概念验证走向生产落地。关键瓶颈在知识维护、幻觉控制、人工兜底。 |
| **获客渠道** | 1. 技术社区开源（本项目已 MIT 开源）；2. 垂直电商社群口碑；3. 与 SCRM/ERP 厂商做集成对接。 |

## 3. 竞品合规性对比总表

| 维度 | MockCliAdapter | NtworkWecomAdapter | 微信客服（WecomKfAdapter） |
|------|----------------|--------------------|--------------------------|
| **传输能力** | 本地模拟，stdout 输出 | 接管真实企微群/单聊 | 官方 1:1 客服对话 |
| **平台支持** | 任意平台（Mac/Linux/Windows） | 仅 Windows + 企微客户端 | 任意平台（需公网回调） |
| **合规性** | 不涉及真实通讯，无风险 | **违反企微协议，有封号风险** | **官方合规** |
| **群聊支持** | 模拟群聊 | 支持真实群聊 | 不支持（仅 1:1） |
| **第三方依赖** | 纯 stdlib | 需 `pip install ntwork` + 企微客户端 | 纯 stdlib（自实现 AES） |
| **实现状态** | **已实现** | **Stub（待 Windows 实机）** | **已实现** |
| **适用场景** | 开发调试 / 离线测试 | 真实群私域维护（高风险） | 合规单聊客服 / 转人工 |

> **结论**：当前主推微信客服（`ADAPTER=kf`）作为合规生产通道；Ntwork 仅作为技术储备，待业务明确需要“进群”且接受封号风险时启用。
