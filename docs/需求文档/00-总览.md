# 需求文档·总览（00-总览）

> 本册是 wecom-group-agent 项目的产品需求文档（PRD）总览，面向将要接手开发或维护的工程师。以下从背景与动机、目标与非目标、明确边界与当前进度、里程碑/路线图四个维度给出全景视图。具体用户分析、功能拆解与非功能需求详见各子分册。技术实现细节请参阅 [技术总览](../技术文档/00-总览.md)。

## 1. 背景与动机

项目初衷是「接管微信群/企微群，做电商带货的私域智能维护」。但从技术调研到务实落地的过程，经历了如下心路：

1. **想接管群对话**：电商私域的核心阵地是微信群/企业微信群。运营希望有一个 Agent 能 7x24 驻群，自动回复客户、维护关系、促复购。
2. **官方无群消息 API**：腾讯不提供任何**实时群消息**的开放接口。微信客服 API 只能 1:1 单聊，进不了群；企微官方 API 同样没有「实时读取群消息并回复」的能力。
3. **只能 hook，但受限极大**：当前业界唯一可行方案是 `ntwork` 注入 Windows 版企微客户端 `WXWork.exe`，通过 DLL 拦截消息。但这有三条硬约束：
   - **仅 Windows**：`ntwork` 依赖 Windows 原生注入机制，Mac（尤其 Apple Silicon）因 SIP + 无逆向工具，完全无法运行。
   - **违反协议**：hook 属于侵入企微客户端，明确违反《企业微信用户协议》，存在**封号风险**（用户已知情）。
   - **不稳定**：企微版本升级可能导致注入失效，需要持续维护。
4. **务实先做合规/agent 这条线**：既然用户主力设备是 Apple Silicon Mac，真实 hook 现阶段跑不通，因此先以**传输无关架构**把 agent 大脑（路由、去重、限流、多轮记忆、LLM 回复、知识库驱动、转人工、防注入）在 Mac 上跑通；同时保留一个未来的合规/真实接入点：
   - **ntwork 企微群 hook**（能进群，但仅 Windows 且违规，封号风险）。

> 一句话总结：**先把"大脑"做扎实，等传输通道条件成熟（Windows 机器）即可无缝接入。**

## 3. 目标与非目标

### 3.1 目标（In Scope）

- 传输无关的架构：Adapter 抽象屏蔽底层差异，核心（Router / Handler / Session）与传输零耦合。
- 群聊与单聊的统一处理：群聊仅在 `@bot` 时触发回复（防刷屏），单聊全回。
- 消息去重：基于 `msg_id` 的滑动窗口去重，防重复推送/网络抖动导致的多回。
- 会话级限流：每个 `chat_id` 两次回复之间最小间隔 2 秒，防轰炸。
- 多轮记忆：按会话维护最近消息上下文，LLM 可感知前文。
- LLM 智能回复：纯 Python 标准库 `urllib` 调用 OpenAI 兼容接口，当前接入 StepFun（阶跃星辰）。
- 知识库驱动：通过 `persona.md`（人设）+ `knowledge.md`（店铺知识）动态组装 system prompt。
- 转人工：遇到投诉、退款、超权限砍价、知识库无法回答的问题时，标记 needs_human 并静默；Router 触发 on_escalate 回调，人工可通过 REST API 接管/释放。
- 持久化会话：SessionStore 支持内存版与 SQLite 持久化版，重启后历史上下文与状态（接管/转人工）可恢复。
- 可配置知识库：支持 static、rag、structured、hybrid 四种知识提供方，通过环境变量 KNOWLEDGE_PROVIDER 切换，无需改代码。
- 零第三方依赖：核心及适配器全部使用 Python 标准库，降低部署与维护成本。
- 可离线测试：Mock CLI 适配器支持交互式与脚本式两种输入，Mac 可全链路跑通。
- 防提示注入：system prompt 要求忽略角色扮演/泄露设定类指令；代码层保证注入内容只作为 `user` 角色进入上下文，绝无可能变成 `system`。

### 3.2 非目标（Out of Scope）

- **不解决"官方无群 API"的根本矛盾**：本项目是受限环境下的务实折中，不是绕过腾讯封控的方案。
- **不跨平台实现 hook**：Mac 上不做企微客户端注入（技术不可行）。
- **不做主动群发/主动推送**：当前仅做被动响应（收到消息才回复），不涉及定时群发、朋友圈、群公告等。
- **不做复杂工作流/工单系统**：转人工已提供标记、静默、REST API 接管/释放；结构化工单系统仍属规划，本阶段不实现。
- **不做语音/图片/视频处理**：当前 `msg_type` 只处理 `"text"`，非文本消息直接忽略。

## 7. 明确边界与当前进度

### 7.1 已验证（Mac 可跑）

- [x] **Mock 全链路跑通**：`MOCK=1` 时，交互式与脚本式输入均能完整走通 `Adapter → Router → Handler → send`。
- [x] **微信客服适配器已落地**：`WecomKfAdapter` 纯 Python 标准库实现回调 HTTP 服务 + msg_signature 验签 + AES 解密 + sync_msg 拉取 + send_msg 下发，零第三方依赖。公网回调联调仍待真机验证。
- [x] **离线 13 项测试全过**：`for f in tests/test_*.py; do python3 "$f"; done` 全绿（LLM、知识库、RAG、结构化 KB、微信客服、接管、SQLite、API、集成、并发、路由、装配层、Web 工作台）。
- [x] **LLM 实测出过真回复**：StepFun（`step-3.7-flash`）实测通过，覆盖维护、询价走知识库、多轮记忆、投诉转人工、防注入等场景。
- [x] **断网降级**：网络断开时正确降级到兜底话术，进程不崩溃。
- [x] **持久化（SQLite）已实现**：`SqliteSessionStore` 支持写穿透落库、重启恢复最近 MAX_CONTEXT 条历史、状态（接管/转人工）恢复。
- [x] **REST API 已实现**：`api_server.py` 提供 health、conversations、messages、takeover、release、queue、agent/preview、config、metrics 等端点，支持 HMAC 恒定时间鉴权。

### 7.2 待实现 / 规划中

- [ ] **ntwork 企微群 hook**：需要 Windows 机器 + 企微客户端扫码登录，当前 [adapters/ntwork_wecom.py](../../adapters/ntwork_wecom.py) 仍为 stub，顶部注释已写明实现要点与字段映射（`conversation_id` / `local_id` / `sender` / `content` / `at` 列表）。
- [ ] **微信客服真机公网回调联调**：`WecomKfAdapter` 代码已完成，需公网可访问地址与腾讯服务器做端到端验证。
- [ ] **结构化工单系统 + 人工工作台前端**：当前仅提供 REST API 契约，前端界面仍属规划。
- [ ] **客户画像/指标事件持久化表**：`Customer`、`MessageRecord`、`MetricEvent` 等数据模型仍属规划。
- [ ] **知识管理后台/配置中心**：知识文件目前为静态 markdown/json，后台管理界面仍属规划。
- [ ] **前端运营后台**：运营配置、监控大盘等前端界面仍属规划，本项目当前仅提供 REST API。

### 7.3 已知限制

- 当前 `msg_type` 只支持 `"text"`，图片/语音/视频/文件被忽略。
- 转人工目前代码层已支持标记 + 静默 + REST API 接管/释放，但结构化工单系统、人工工作台前端仍属规划。
- 限流为单进程内存实现，多进程部署时各进程独立计数（不影响功能，但可能短暂超发）。
- .env.example 已补全 `KNOWLEDGE_PROVIDER`、`STORE`、`SQLITE_PATH`、`ADMIN_TOKEN`、`API_HOST`/`API_PORT`、`LLM_EMBED_*` 等配置项示例。

## 8. 里程碑 / 路线图

> **MVP 决策（2026-07-10，双 AI 评审共识，详见 [DISCUSSION.md](../../DISCUSSION.md) #1/#2 与 [产品化路线图](../产品化路线图.md)）**：
> **微信客服官方 1:1 通道 = MVP 唯一真渠道**；ntwork 企微群 hook 降级为远期 P2（合规灰色 + 仅 Windows）。私域电商可通过视频号/公众号/小程序客服入口 + 群内置顶引导，将流量导至合规 1:1 单聊，不损失场景覆盖。

| 阶段 | 目标 | 交付物 | 状态 / 验收标准 |
|------|------|--------|----------|
| **M0** | Mac 上跑通全链路 | 传输无关架构 + Mock CLI + EchoHandler + LLMHandler + 离线测试 | ✅ 完成：13 套离线测试全绿；demo_agent.py 17 场景回复质量可接受 |
| **M1（MVP 主线）** | 接入合规通道 | `WecomKfAdapter`（`ADAPTER=kf`）+ [真机联调 SOP](../技术文档/07-真机联调SOP.md) | 代码/18 项测试/SOP 就绪；待公网资源做端到端验证——官方后台配置回调后 1:1 自动回复；不封号 |
| **M3（生产加固）** | 敢用、可运维 | Docker 三服务 + 启动自检 + evals 金标评测 + Web 工作台 + SQLite 持久化 + REST API | ✅ 大部分完成（2026-07-10）：检索层 17/17 离线、回复层 21/21 在线；余项见 [产品化路线图](../产品化路线图.md) P1/P2 |
| **M2（降级远期）** | 群聊 hook（Windows） | 实现 `NtworkWecomAdapter.start/send` | 📋 P2：MVP 决策降级；如业务强依赖群，需 Windows 实机 + 充分风险沟通 |
| **M4** | 扩展能力 | 富文本（图片/链接卡片）+ 主动运营（欢迎语/活动提醒）+ 结构化工单 + 客户画像 | 📋 规划 |

> **风险预警**：M2 的 ntwork 方案存在封号风险，建议作为"灰度试探"而非"全量主力"。M1 的微信客服是合规主力，但受限于只能 1:1。若业务核心在群维护，则 M2 是必经之路，需与运营充分沟通风险并准备备用号/备用方案。
