# 08 · 评测与质量门禁

> 本文档描述 `evals/` 评测体系与合规换版门禁的设计、用法与扩展方式。
> 关联源码:[`evals/run_evals.py`](../../evals/run_evals.py) · [`evals/golden.json`](../../evals/golden.json) · [`.github/workflows/compliance-gate.yml`](../../.github/workflows/compliance-gate.yml)

## 1. 为什么需要评测体系

单元测试回答"代码是否按写的运行",评测回答"**agent 是否按产品预期回答**"。二者缺一不可:

- 换模型(step-3.7-flash → 其它)、改 persona、改知识库,单元测试全绿也可能让回复质量/合规性悄悄劣化;
- 转人工的**误召**(不该转的转了=浪费人力)与**漏召**(该转的没转=客诉),必须双向可回归;
- 合规红线(私下交易/功效宣称/让价承诺/竞品贬低等)一旦破线是硬风险,不能靠人工抽查。

## 2. 三层评测架构

| 层 | 命令 | 依赖 | 跑什么 | 何时跑 |
|---|---|---|---|---|
| **离线层** | `make eval` | 无网络、无密钥 | 知识检索断言 + 转人工**规则层**准召(确定性) | 每次 push(CI `test` job) |
| **在线层** | `make eval-online` | `.env` 的 `LLM_API_KEY` | 离线层 + 真实 LLM 回复断言 + session 旗标核对 | 改动 prompt/知识/判定逻辑后手动 |
| **门禁层** | `make gate` | 同上 | 只跑**合规红线+转人工**类目,强制在线+LLM-as-judge,**0 失败 0 跳过才放行**(exit 1 拦截) | 换模型/改 persona **必跑**;CI 侧 `compliance-gate.yml` 自动触发 |

### 2.1 离线层如何"真测 RAG"——确定性本地 embedding

难点:RAG 检索链路(切块→向量→余弦→三路召回→并集)依赖 embedding API,CI 无网络。
只测"API 失败回退全量"分支等于没测检索。

方案:评测运行器注入 `_local_embed`——字符 bigram 经 md5 哈希装进 4096 桶计数当向量。
它不衡量真实语义,但**中文 FAQ 场景下 bigram 重叠与相关性高度相关**,足以离线驱动完整检索链路且跨平台可复现(md5 稳定)。三重隔离保证不污染生产:

1. `RagKnowledgeProvider` 的 `embed_fn` 注入路径**设计上不写磁盘缓存**(内存缓存,见 `core/knowledge.py` retrieve);
2. 评测传独立 `embed_model="eval-local-bigram-4096"`,缓存键天然错开(双保险);
3. `small_kb_max=1` 强制走检索(样例知识库在默认阈值下会直通全量,测不到检索)。

### 2.2 转人工准召的双向断言

金标案例的 `escalation` 声明:

```json
{"escalation": {"expect": true,  "label": "售后退款/理赔"}}   // 漏召方向:必须触发且标签一致
{"escalation": {"expect": false}}                            // 误召方向:规则层不得触发
```

- **离线**:直接调 `core.llm_handler.classify_escalation`(规则层,确定性)断言;标签值经 `_KNOWN_LABELS` 与 core 常量同源校验,金标手滑写错标签会当场报错(防文案漂移);
- **在线**:跑完真实 `LLMHandler.reply` 后核对 `session.needs_human` 与 `escalation_reason` 前缀(reason 格式为`标签:命中「关键词」`);
- `expect: true` 但 `label` 为兜底标签(`AI 判定复杂场景`)的案例,规则层允许不触发(该类靠 LLM 兜底,离线只在线验)。

### 2.3 门禁层与 LLM-as-judge

子串断言有天生盲区:**否定句**。真实案例:合规回复"真没法打包票**保证明天**必到"被 `must_not_include: ["保证明天"]` 误杀(见 09 难点记录)。因此:

- 子串断言只做"**必须出现拒绝措辞**"方向(`must_include_any: ["没法", "不敢", ...]`);
- "**不得出现肯定式承诺**"方向交给 judge:`_judge_chat` 把七条红线(私下交易/功效宣称/超知识承诺/理赔承诺/到货承诺/贬低竞品/未成年促销)交给同一 LLM 做语义审查,输出 violations 数组;
- judge 自身不可用(网络/解析失败)时**保守拦下**报"需人工复核",不放行。

CI 侧 `compliance-gate.yml` 只在 `prompts/**`、`core/llm_handler.py`、`evals/**` 变更时触发;
仓库未配置 `LLM_API_KEY` secret 时优雅跳过并提示本地跑 `make gate`(配置 secret 后即全自动)。

## 3. 金标集(golden.json)案例结构

```json
{
  "id": "唯一 ID(--only 按子串过滤)",
  "category": "结构化查价 | 库存状态 | RAG 检索 | 主链路 hybrid | 静态全量 | 合规红线 | 转人工",
  "provider": "static|rag|structured|hybrid|null(null=无检索断言)",
  "query": "客户原话",
  "retrieval": {"must_include": [], "must_not_include": []},
  "escalation": {"expect": true, "label": "标签"},
  "reply": {"must_include_any": [], "must_not_include": []},
  "notes": "为什么这么断言(含已知陷阱说明)"
}
```

当前规模:**35 案**(检索类 18 · 转人工 12 · 合规红线 5;门禁类目=后两者共 17),离线覆盖 30、在线全覆盖。

### 3.1 写好断言的三条经验(实测踩坑总结)

1. **must_not 只断言设计保证的事**:RAG 语义 top-k 加相对阈值前,零分 chunk 会凑满坑位,断"不该混入"必炸;结构化克数通路是精确集合交集,可以放心断 must_not。
2. **警惕否定句**:`must_not_include: ["保证明天"]` 会误杀"没法保证明天到"。肯定式违规交 judge。
3. **在线断言要跑两遍再合入**:LLM 非确定性,单跑通过≠稳定通过(本项目约定:转人工/合规类新案例双跑全绿才算数)。

## 4. 如何扩展

- **加案例**:编辑 `golden.json`,先 `python3 evals/run_evals.py --only 新id` 离线验,再 `--online --only 新id` 双跑;
- **加转人工标签**:改 `core/llm_handler.py` 的 `REASON_*` 常量与 `_ESCALATION_RULES`,运行器的 `_KNOWN_LABELS` 同步(同一 import,漂移会被校验拦住);
- **换模型**:改 `.env` 的 `LLM_MODEL` → `make gate` 全绿才允许上线(这是门禁存在的全部意义)。
