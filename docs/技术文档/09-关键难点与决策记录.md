# 09 · 关键难点与决策记录

> 本文档是产品/技术/测试三侧的**难点与决策日志**:记录"为什么这么做、当时卡在哪、怎么验证的"。
> 新难点按时间追加,不删旧记录(错误的决策也保留并标注,防止重犯)。
> 时间基准:2026-07-08 接管 ~ 2026-07-10 路线图 P0/P1/P2 完工。

## A. 产品侧

### A1. MVP 渠道决策:微信客服官方通道,群 hook 降级远期

- **难点**:私域电商的直觉需求是"接管微信群",但 ntwork 群 hook 仅 Windows、违反企微协议有封号风险;微信客服官方通道合规但只能 1:1。
- **决策**(双 AI 共识,DISCUSSION #1/#2):**微信客服 = MVP 唯一真渠道**,群 hook 降 P2 远期。
- **理由**:合规无封号风险;加解密链路已就绪可公网部署;微信客服可挂视频号/公众号/小程序客服入口——私域电商本就在腾讯生态内,合规通道不损失场景覆盖。

### A2. 转人工的误召/漏召不对称成本与三层联动

- **难点**:漏召 = 客诉(该转的没转),误召 = 1-2 人小团队的收件箱被噪音淹没。二者成本都高但方向相反,单靠 prompt 无法双向保证。
- **决策**:**规则层硬触发(保漏召)+ persona 判断策略(控误召)+ 知识库答案供给(降误召)三层联动**:
  - 规则层:客户点名人工/售后损害词/情绪词命中即确定性转人工,LLM 忘打标记也不漏;
  - persona 层:政策**咨询**("拆封了还能退吗")与实际**诉求**("我要退款")分野;普通砍价先标准话术、坚持才转;
  - 知识层:高频售后问题(如"退款几天到账")必须有答案条目,否则 LLM 按"知识答不了→转人工"条款正确地烧掉人工。
- **实证**:三层各修过一次真实误召(见[测试文档/06](../测试文档/06-评测与测试记录.md)记录)——**误召要三层都对才降得下来**,这是本期最重要的产品认知。

### A3. 政策名词 ≠ 售后诉求(#13 探针)

- **难点**:"退款/退货"是政策名词,出现在句子里既可能是咨询("怎么退货呀")也可能是诉求("我要退款")。关键词一刀切会把 persona 刚修好的误召确定性地引回来。
- **决策**:售后关键词拆两档——损害/纠纷词(赔偿/破损/发霉/吃出…)无歧义,命中即硬触发;政策名词须**同现诉求标记**(我要/给我/怎么还没…)才触发,纯咨询交 LLM。
- **代价与接受**:"给我退了吧"(有诉求无名词)规则层漏召,落 LLM 兜底——符合"宁漏给 LLM(在线有 persona 兜底),不误伤咨询(误召无兜底)"的取向。

### A4. `escalation_reason` 格式约定:`标签:命中「关键词」`

- **决策**(DISCUSSION #10):冒号前缀可扫可聚合——运营者在工作台收件箱一眼扫标签;`/api/metrics` 的 `escalation_by_reason` 按前缀聚合零额外接口。标签集固化为 `core/llm_handler.py` 顶部常量,评测断言 import 同一常量防文案漂移。

### A5. 合规红线是硬约束,优先级高于促成交

- **决策**:persona 明文七条(私下交易/功效宣称/超知识承诺/理赔承诺/到货承诺/贬低竞品/未成年),且**换模型/改 persona 必须过 `make gate` 全绿**(0 失败 0 跳过)才允许上线。合规不靠自觉,靠门禁。

### A6. 回复"细节质量"也要机制化,不能靠感觉(真机首聊反推)

- **难点**:上线首聊即暴露 4 个细节硌手点(markdown 泄漏成星号/复读开场白/emoji 每条出现/敷衍语被曲解)——功能测试全绿也拦不住"不好用"。
- **决策**:①**格式即产品**:微信不渲染 markdown,persona 立纯文本铁则+排版模板锚定;②细节规则(延续性/emoji 频率/敷衍语)全部写进 persona 并配**多轮评测案例**锁定——细节问题大多只在多轮暴露,运行器为此支持 turns 字段;③修复必须真机重启后由用户复测体感闭环。

## B. 技术侧

### B1. Content-Length 负值家族(同一漏洞,三处三个暴露面)

- **难点**:`int(headers.get("Content-Length") or 0)` 不拦负数,`rfile.read(-1)` 会阻塞读到 EOF——处理线程卡死。同一模式存在于 admin_console(内网)、api_server(回环)、**wecom_kf 回调(9000,唯一真公网面)**三处,且回调口的 body 读取在验签之前,未授权可触发。
- **修法**:`max(0, int(...))` + `except ValueError` 归 0;归 0 后 body 为空→验签失败回 500,对腾讯真实回调无影响。
- **教训**:review 发现一处此类问题时,**必须全库 grep 同款模式**——本例从 console review 顺藤摸瓜找到公网面上的同款,严重度远高于原发现。

### B2. access_token 跨进程共享

- **难点**:agent 与 api_server 双进程各自缓存 access_token,一方 gettoken 会使另一方的失效(腾讯同 secret 新令旧亡),互踢死循环。
- **修法**:`WECOM_TOKEN_FILE` 文件共享(带锁,失效主动清缓存重取);compose 双服务挂同一数据卷。P2-8 的 `AGENT_STATS_FILE` 沿用同一模式。

### B3. 回调"处理失败回 500"防丢消息

- **决策**:微信客服回调处理失败**不回 success 而回 500**,让腾讯重试;cursor 不推进,重试会重新拉取——宁可重复处理(有 msg_id 去重兜底),不可静默丢客户消息。

### B4. RAG top-k 无阈值的"凑数 chunk"

- **难点**:语义通路取 top-k 不看分数,零分 chunk 也凑满 k 坑位。小库无害(多带上下文),但使评测无法断言选择性,规模化后放大噪音。
- **修法**(P2-7c):top-k 内再按 `分数>0 且 ≥top1×RAG_SCORE_RATIO`(默认 0.6,0=关闭)过滤;全体皆噪声时语义通路自然为空,由关键词/常驻/全文回退三防线兜底,行为不劣化。
- **验证**:当初因凑数被迫撤掉的两条选择性断言(税号/顺丰不得混入)加回后通过。

### B5. 规格标准化:克数通路

- **难点**:客户口语与商品数据的单位写法不一致("半斤"↔"250g"、全角"２５０ｇ"、"0.5 千克"),子串匹配召回≈0。
- **修法**(P2-7b):NFKC 归一 + 重量表达提取(半斤/N斤/N两/N克/Ng/N千克/Nkg)归一克数集合求交集,作为 `_is_hit` 子串外的第二通路。集合交集是精确匹配,评测敢断 must_not。

### B6. 纯 stdlib 约束下的取舍

- AES-256-CBC + PKCS7 + SHA1 验签纯 Python 自实现,FIPS-197 官方向量钉住正确性——零第三方依赖使 Docker 镜像无 pip 层、供应链面为零,代价是需要向量测试保证密码学实现不出错。
- LLM-as-judge、embedding、企微 API 全走 `urllib`,统一"出错必兜底、绝不裸抛"。

### B7. 双 AI 共享工作区的协作纪律

- **难点**:两个 Claude 会话共享同一目录与 `.git`,并发提交/工作区互踩风险高。
- **纪律**(全程零冲突的原因):① 只 `git add` 自己认领的文件,严禁 `-A`;② 认领文件写进 DISCUSSION 楼层,改对方边界前打招呼;③ 交付贴 commit hash,对方 review 回帖(LGTM 或改进点);④ 看到对方未提交的在制品,绕开不碰。
- **实证**:双方 40+ 次提交交错零冲突;互审抓出对方(与自己)的真实缺陷十余处。

### B8. 单连接约束下的跨进程发送:outbox 文件队列桥

- **难点**:智能机器人同一 bot 全网只允许一条长连接——api_server 进程永远发不了消息,后台人工回复只能 `sent=False`;而把 api 并进 agent 进程又违背双进程部署形态。
- **方案**:outbox 文件队列(与 token/stats 文件同一"共享文件"模式):无 ws 进程 `send()`→追加 jsonl;在线 agent 每轮询节拍 `os.replace` 原子改名消费(追加竞争窗口的新条目落新文件,不丢不重),复用 `send()` 真实分支使 24h 窗口校验天然生效。
- **重放边界(review 抓出的缺口)**:drain 必须查条目时效——超 600s 丢弃,否则 agent 停机重启会把昨天的人工回复发进今天的语境。
- **附带教训**:双实例互踢风暴真实上演过一次(45009 限频→指数退避→自愈)——**起第二个 agent 前先确认场上没有活的**。

### B9. 跨层数据口径要在写入点统一,不能靠显示层兼容

- **事故**:消息 timestamp 三处三个口径——适配器毫秒、Router/api 造消息**默认 0**(初报为"秒",查实更糟);console 的 `>1e12` 兼容判断让显示一直正常,**掩盖了数据层的病**,直到要做按时聚合才暴露。
- **规则**:数据口径在**写入点**统一(所有 Message 构造处显式毫秒),显示层兼容只作过渡;发现口径不一致时先查实际存储值,别信目测换算。

## C. 测试侧

### C1. 子串断言的否定句盲区

- **事故**:合规回复"真没法打包票**保证明天**必到"(完全合规)被 `must_not_include: ["保证明天"]` 误杀——门禁首跑即误拦。
- **规则**:子串 must_not 只用于无歧义违规短语;"不得出现肯定式承诺"这类**否定语义**方向一律交 LLM-as-judge。

### C2. 离线测 RAG 的确定性 embedding

- 见 [10-评测与质量门禁](10-评测与质量门禁.md) §2.1。要点:bigram 哈希装桶(4096)驱动真实检索链路,md5 保证跨平台可复现;三重隔离(embed_fn 不落盘 + 独立缓存键 + small_kb_max=1)保证不污染生产缓存。

### C3. 在线评测的非确定性纪律

- LLM 输出非确定,单跑通过≠稳定。**约定:转人工/合规类新案例双跑全绿才算合入**;评测失败先读回复原文分辨"断言错还是行为错"(本期两类都出现过,处置完全不同)。

### C4. transport 注入测试法

- `LLMHandler(transport=fake)` 短路真实网络与 key 校验,离线可测全链路(消息组装/历史窗口/标记剥离/兜底);`ConsoleApp(fetcher=...)`、`WecomKfAdapter(http_get_json=...)` 同款模式。**可注入的旁路是纯 stdlib 项目可测性的生命线**。

### C5. 样例数据当标杆

- `prompts/products.json` 纳入"零 schema 问题"测试(`test_validate_products_file_real_sample`)——样例即模板,用户照着改,样例本身必须永远干净。

### C6. 评测抓真问题的时间线

- 完整记录见 [测试文档/06-评测与测试记录](../测试文档/06-评测与测试记录.md)。摘要:在线评测先后抓出 **2 个 persona 误召**("拆封了还能退吗"被转人工/"再便宜点"被转人工)、**1 个知识缺口**("退款几天到账"无答案条目)、**1 个断言误伤**(否定句)、**1 个规则层回归**(#13 政策名词误伤)——每一轮失败都换来一层修正,这套体系第一天就回本。

### C7. 细节类回复问题只有多轮才暴露

- 复读开场白/上下文断裂这类问题,单轮案例永远测不出。评测运行器支持 `turns`(前几轮客户消息**真实过模型**、bot 回复进 history,模拟 Router 行为),最后一轮断言——真机首聊的 4 个细节问题全部由此转化为可回归案例。

### C8. 护栏必须先证明"真的会红"

- 内容护栏(persona 机制不可删/两库价格一致性等)上线前做**变异冒烟**:故意改坏(¥39→¥38)看它红、还原看它绿。没红过的护栏等于没测过的护栏——它可能永远绿是因为它什么都没查。gate 的单案例重试同理:重试必须**上限 1 次+日志列明首跑问题**,否则重试就是洗绿机。
