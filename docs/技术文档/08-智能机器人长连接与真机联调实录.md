## 1. 产品决策:为什么从「微信客服」转向「智能机器人」先落地

原 MVP 计划(路线图 P1-2)走微信客服公网回调。真机联调启动时发现用户实际条件:

| 现实条件 | 对原计划的影响 |
|---|---|
| 企业「哈喽咸鱼」**未认证** | 微信客服对外接待微信用户受限,认证前该通道无法服务真实外部客户 |
| 用户在企微后台实际打开的是「**智能机器人**」创建页 | 该产品(2025 起开放)提供**长连接模式:无需域名/IP/加解密** |
| 本机无固定公网 IP | 微信客服必须公网 HTTPS(隧道已备好但多一层依赖);长连接零公网依赖 |

**决策(PM 视角)**:双通道策略——
- **智能机器人(ADAPTER=aibot,本册)**:立即落地。服务**企业内部成员**(单聊 + 群聊@bot),官方 API、合规、零公网依赖。作用:① 真渠道端到端验证 AI 大脑;② 内部导购助手/员工查价工具;③ **群聊场景以合规方式部分回归**(项目最初诉求)。
- **微信客服(ADAPTER=kf,见 04/07 册)**:企业认证完成后上线,服务**外部微信客户**。代码/测试/联调 SOP 全部就绪,只等认证。

## 2. 协议(实测核对过的最小集)

```
wss://openws.work.weixin.qq.com          ← WebSocket,免消息加解密
├─ 鉴权: {"cmd":"aibot_subscribe","headers":{"req_id":R},"body":{"bot_id":B,"secret":S}}
│        → {"errcode":0} 即订阅成功;同一机器人仅允许一条长连接,新连踢旧连
├─ 心跳: {"cmd":"ping","headers":{"req_id":R}} 每 ~30s;长时间无心跳服务端断开
├─ 收消息: {"cmd":"aibot_msg_callback","headers":{"req_id":R},"body":{
│           msgid, aibotid, chatid(仅群聊), chattype:"single|group",
│           from:{userid}, msgtype:"text", text:{content:"@bot名 正文"}}}
├─ 回复:  {"cmd":"aibot_respond_msg","headers":{"req_id":R ←必须透传回调的},"body":{
│           msgtype:"stream", stream:{id:唯一, finish:true, content:全文}}}
│         (finish=true 一次成型 = 非流式;真流式则同 stream.id 多次推送)
└─ 事件:  aibot_event_callback(enter_chat/模板卡片/feedback/disconnected_event)
```

约束:回复窗口 = 收到回调后 24h;单会话限 30 条/分钟、1000 条/小时;流式从首发起 10 分钟内必须 finish;群聊只有 @机器人 才回调(天然符合 Router 的群聊触发语义)。

## 3. 实现设计

- **WebSocket 客户端纯标准库自实现**(`_WSClient`,~100 行):HTTP Upgrade 握手(校验 `Sec-WebSocket-Accept` = SHA1(key+GUID))、RFC 6455 帧编解码(`_encode_frame`/`_read_frame` 模块级纯函数,可离线单测)、客户端强制掩码、协议层 ping 自动回 pong、分片消息组装、4MB 帧上限防失配。**不引第三方 websocket 库**——与 wecom_crypto 自实现 AES-256 同一传统。
- **心跳节拍不开线程**:socket 设 5s 短超时,recv 超时即检查距上次 ping 是否 ≥30s,是则发心跳——单线程收发环,省一个心跳线程的同步复杂度。
- **断线自愈**:指数退避重连(1s→2s→…→30s 封顶),订阅成功即复位;`disconnected_event`(被新连接踢)按断线路径走。
- **被动回复的 req_id 上下文**:`aibot_respond_msg` 必须透传**回调的** req_id → 适配器记 `chat_id → 最近回调 req_id` 映射,`send()` 取用;无上下文(如人工凭空发起)打日志不抛错(主动推送 aibot_send_msg 留待需要时补)。
- **消息映射**:单聊无 chatid → 以 `aibot-single-{userid}` 为会话键(同一用户稳定同一会话);群聊 `is_at_bot=True` + 剥落 `@机器人名 ` 前缀再喂 LLM。
- **装配与自检**:`main.py` 增 `ADAPTER=aibot` 分支;`check_config` 缺 `WECOM_AIBOT_ID/SECRET` 拒绝启动(P0 自检口径)。

## 4. 关键困难点(按时间序,含排查过程)

| # | 困难 | 现象 | 根因与解法 |
|---|------|------|-----------|
| 1 | **产品歧义:智能机器人 ≠ 微信客服** | 用户按引导去开通道,截图却是「创建智能机器人」页 | 企微后台两个入口名字都像"机器人"。识别出这是不同产品后**顺势改道**(见 §1),而不是硬掰回原计划——未认证企业走微信客服本来就会被卡 |
| 2 | **凭据手抄必错(errcode 853000)** | 首次订阅 `invalid bot_id or secret`;截图 OCR 的 Bot ID 两种拼写(taiby…/aiby…)都被拒 | 40+ 位大小写混合串里 `I/l/1` 不可分辨;且**保存后 Bot ID 与创建页展示的完全不同**(aibduy10…)。解法:让用户用页面「复制」按钮取值。**教训:凭据永远走复制,不走截图** |
| 3 | **文档抓取三连卡** | WebFetch 工具月配额 402;GitHub release 直连 curl 超时;官方文档页正文藏在 JS 数据块 | ① 改本地 curl;② 走本机 Shadowrocket socks5(127.0.0.1:1082)下载 cloudflared;③ 对 HTML 里的转义 JSON 做 unescape+剥标签后按关键词(aibot_subscribe/心跳机制说明)定位抽取,拿到全部帧格式原文 |
| 4 | **隧道预检的取舍** | 为微信客服备的 cloudflared quick tunnel 建到一半发现改道 aibot 后用不上 | 保留验证结论(公网链路可用,502=边缘通、本地未起服务)写入 #16;微信客服认证后直接复用该路径,预检不白做 |
| 5 | **零依赖 vs 官方 SDK** | 官方给的是 Node.js/Python SDK(aibot-python-sdk) | 长连接协议本身只是「WebSocket + 明文 JSON」,自实现 ~100 行即覆盖;免加解密比微信客服通道还简单。坚持零依赖,部署面不变 |

## 6. 运维要点

- 启动:`ADAPTER=aibot HANDLER=llm KNOWLEDGE_PROVIDER=hybrid STORE=sqlite python3 main.py`
- `.env`:`WECOM_AIBOT_ID`/`WECOM_AIBOT_SECRET`(后台「智能机器人」API 配置区**复制**取值;Secret 泄露风险时在后台「随机获取」轮换并同步 .env)
- 同一机器人只允许一条长连接——**⚠️ 双实例互踢风暴**:同一 bot_id 起两个进程(如 compose 里的 agent + 本地调试进程)会以 ≤30s 周期无限互踢(disconnected_event ↔ 重连),现象是两边日志都在"订阅成功→被踢下线"循环。多副本部署必须主备,本地调试前先停生产进程;
- 与 api_server/console 组合:同 `STORE=sqlite` 同库即可,人工回复经 `adapter.send`(需 24h 内该会话有过用户消息)。
