## 1. Adapter 抽象

源码：[core/adapter.py](../../core/adapter.py)

`Adapter` 是传输无关架构的最底层，负责把「具体传输」（Mock CLI / 企微 hook / 微信客服回调）的消息收发映射到统一的 [`Message`](../../core/message.py) 模型。核心（Router、Handler、SessionStore）只依赖这个抽象，换传输不动核心。

### 1.1 start / send 契约

```python
class Adapter(ABC):
    @abstractmethod
    def start(self, on_message: OnMessage) -> None:
        """开始接收消息；每收到一条就调用 on_message(Message)。"""
        raise NotImplementedError

    @abstractmethod
    def send(self, chat_id: str, text: str) -> None:
        """往指定会话发一条文本消息。"""
        raise NotImplementedError
```

- `start` 是阻塞入口：实现方启动自己的事件循环（如 mock 的 `sys.stdin` 循环、ntwork 的回调循环），每收到一条消息就调用 `on_message`。
- `send` 是主动发送：只传 `chat_id` 和 `text`，实现方负责把文本送到对应的群或单聊。
- `OnMessage = Callable[[Message], None]`，即 [`Router.on_message`](../../core/router.py) 的签名。

### 1.2 如何新增一个适配器

1. 在 `adapters/` 下新建文件，继承 `Adapter`；
2. 实现 `start` 与 `send`；
3. 在 [`main.py`](../../main.py) 的 `build_adapter()` 中按环境变量注册；
4. Router、Handler、SessionStore 全部复用，无需改动核心。

## 2. MockCliAdapter

源码：[adapters/mock_cli.py](../../adapters/mock_cli.py)

本地可运行的模拟适配器，用于在 Mac 等无 hook 环境把「消息进 → 路由 → 回复 → 发出」全链路跑通。`send()` 直接打印到 stdout，模拟发消息成功。

### 2.1 两种输入模式

| 模式 | 触发条件 | 输入格式 | 用途 |
|---|---|---|---|
| **交互模式** | 默认（未设 `MOCK_SCRIPT`） | `chat_id\|sender\|content` | 本地手动调试 |
| **脚本模式** | `MOCK_SCRIPT=xxx.json` | JSON 数组，每条为 `Message` 字段的字典 | 自动化验证去重/限流/触发策略 |

**交互模式示例**：`群A|张三|@bot 你好`
- 内容里含 `@bot` 视为 @ 了机器人，该 token 会被剥掉，且 `is_at_bot=True`。

**脚本模式字典**：可含 `chat_id`、`chat_type`、`sender_id`、`sender_name`、`content`、`msg_type`、`is_at_bot`、`timestamp` 等任意字段。缺省默认值为 `chat_id="群A"`、`chat_type="group"`、`sender_name="某人"`、`msg_type="text"`、`msg_id="script-{idx}"`。

### 2.2 消息如何构造

- **交互模式**：`msg_id` 使用自增序号 `cli-{seq}`；`chat_type` 固定为 `"group"`；`sender_id` 与 `sender_name` 均取输入的 sender；`is_at_bot` 由内容是否含 `@bot` 决定。
- **脚本模式**：`msg_id` 优先取字典中的值，否则生成 `script-{idx}`；`is_at_bot` 优先取字典显式值，否则检查 `content` 中是否含 `@bot`。

## 3. NtworkWecomAdapter

源码：[adapters/ntwork_wecom.py](../../adapters/ntwork_wecom.py)

真实接管企业微信群（或单聊）的 hook 适配器。

### 3.1 为何仅 Windows

- `ntwork` 通过注入 Windows 版企微客户端 `WXWork.exe` 获取消息。
- Mac 因 SIP 保护且缺乏对应逆向工具，无法注入，因此该适配器**只能在 Windows 上运行**。

### 3.2 当前状态：Stub

本阶段未实现。`start` 与 `send` 均直接抛出 `NotImplementedError`：

```
ntwork 真实 hook 适配器尚未实现，且只能在 Windows + 企微客户端上运行。
本地验证请用 MOCK=1 走 MockCliAdapter。实现要点见本文件顶部注释。
```

### 3.3 Windows 上实现要点与字段映射

文件顶部注释已给出完整实现框架，核心如下：

**1. 启动**
```python
import ntwork
wework = ntwork.WeWork()
wework.open()          # 拉起/附着企微客户端，扫码登录
@wework.msg_register(ntwork.MT_RECV_TEXT_MSG)
def _on(wework, event):
    msg = _map_event(event)
    on_message(msg)
wework.wait_login()  # 进事件循环阻塞
```

**2. 字段映射**（以 ntwork 事件 `data` 为准，需实机核对）

| Message 字段 | ntwork event data 来源 | 说明 |
|---|---|---|
| `chat_id` | `data["conversation_id"]` | 群/会话唯一 id |
| `chat_type` | `"group"` if 群会话 else `"single"` | 需判断会话类型 |
| `msg_id` | `data["local_id"]` / `data["msgid"]` | 用于去重 |
| `sender_id` | `data["sender"]` | 发送者标识 |
| `sender_name` | 通讯录/群成员查询补全 | ntwork 可能只给 id |
| `content` | `data["content"]` | 纯文本内容 |
| `is_at_bot` | 解析 `content` 或 at 列表 | 判断是否 @ 机器人自己 |

**3. 发送**
```python
wework.send_text(conversation_id=chat_id, content=text)
```

### 3.4 版本锁定 / 关闭自动更新的坑

企微客户端会频繁自动更新，更新后注入接口可能变更，导致 hook 失效。生产环境必须：

- **关闭自动更新**：防止客户端在夜间静默升级；
- **锁定可注入版本**：每次升级后需重新验证 `ntwork` 的接口兼容性，确认稳定后再允许升级。

## 4. 微信客服适配器（已实现）

源码：[adapters/wecom_kf.py](../../adapters/wecom_kf.py)、[adapters/wecom_crypto.py](../../adapters/wecom_crypto.py)

官方合规的 1:1 对话通道，不进群，适合「私域单聊维护」或「转人工客服」场景。`ADAPTER=kf` 已接入 [`main.py`](../../main.py)。

### 4.1 已实现的核心能力

- **回调 HTTP 服务**：基于 `socketserver.TCPServer` + `http.server.BaseHTTPRequestHandler` 启动回调服务器，接收企微服务器的 GET（URL 验证）与 POST（事件通知）请求。
- **签名验证 + AES 解密**：`adapters/wecom_crypto.py` 提供【纯 Python 自实现 AES-256-CBC + PKCS#7(block_size=32) + SHA1 签名 + 打包/解包】，等价企业微信官方 `WXBizMsgCrypt`，零第三方依赖。用 FIPS-197 标准向量钉住 AES 正确性。密文长度非法 / 签名错误 / receiveid 不符统一抛 `WeComCryptError`。
- **sync_msg 拉取消息**：回调只告知「有消息」，主动调用 `sync_msg` 分页拉取（最多 20 页兜底），校验 `errcode`；错误时不把错误响应当成「无消息」，避免静默丢消息。
- **send_msg 下发消息**：调用 `kf/send_msg` 向用户发送文本回复。
- **access_token 带过期缓存**：获取后按 `expires_in` 缓存，过期前 60 秒自动刷新。
- **HTTP 客户端可注入**：`http_get_json` / `http_post_json` 支持注入，便于离线测试。
- **缺 key 友好报错**：`WECOM_ENCODING_AES_KEY` 为空时构造 `WecomKfAdapter` 抛出带字段名的 `RuntimeError`，而非晦涩的 crypto 报错。

### 4.2 回调处理流程

```
企微服务器 ──POST──► 回调 HTTP 服务器
                      │
                      ├─ 验证 msg_signature（SHA1 排序签名）
                      ├─ AES-256-CBC 解密消息体
                      ├─ 提取 Token / OpenKfId
                      ├─ sync_msg 分页拉取完整消息列表
                      └─ 每条消息映射为 Message 后投递 on_message
```

### 4.3 与官方加解密库的等价性

`adapters/wecom_crypto.py` 中的 `WXBizMsgCrypt` 与腾讯官方 `WXBizMsgCrypt` 算法等价：

| 环节 | 实现细节 |
|---|---|
| 密钥 | `EncodingAESKey` base64 解码 → 32 字节 AESKey |
| 算法 | AES-256-CBC，IV = AESKey 前 16 字节 |
| 打包 | random16 + msg_len(4 字节大端) + msg_bytes + receiveid_bytes |
| 填充 | PKCS#7，block_size = 32（腾讯规范） |
| 签名 | SHA1(sorted([token, timestamp, nonce, encrypt])).hexdigest |

**离线测试**：[tests/test_kf_offline.py](../../tests/test_kf_offline.py) 覆盖 FIPS-197 AES 向量、PKCS#7 填充、`encrypt_msg ↔ decrypt_msg` 往返、签名篡改检测、URL 验证、完整 POST 回调链路、HTTP 异常降级、缺 AES key 报错、密文长度非法、errcode 非 0 等 12 个场景。

### 4.4 环境变量

```
ADAPTER=kf
WECOM_CORP_ID=
WECOM_KF_SECRET=
WECOM_CALLBACK_TOKEN=
WECOM_ENCODING_AES_KEY=
WECOM_CALLBACK_PORT=9000
WECOM_CALLBACK_PATH=/wecom/callback
```

### 4.5 当前状态

`main.py` 的 `build_adapter()` 已按 `ADAPTER=kf` 装配 `WecomKfAdapter()`。真机与腾讯服务器互通尚待公网回调联调。

## 5. 部署形态

### 5.1 核心可部署在任意机器

Agent 核心（Router + Handler + SessionStore）纯 Python 标准库，零第三方依赖，可部署在：

- 本地 Mac / Linux / Windows 开发机
- 云服务器（ECS / 轻量应用服务器 / Docker 容器）

### 5.2 部署形态差异

- **Mock 调试**：当前在 Apple Silicon (M5) Mac 上已通过 `MOCK=1` 跑通全链路。
- **NtworkWecomAdapter**：必须与 `WXWork.exe` 同机运行，即需要一台 Windows 机器（物理机或云桌面）常驻登录企微客户端。
- **微信客服（WecomKfAdapter）**：核心可部署在任意有公网 IP 的服务器上，无需绑定特定操作系统；需要可被企微服务器访问的公网回调地址（或内网穿透）。

### 5.3 推荐拓扑

```
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│   用户设备       │     │  Windows 机器    │     │  云服务器/本机    │
│  (企微客户端)    │◄───►│  WXWork.exe     │     │  agent 核心      │
│                 │hook │  + ntwork hook  │◄────│  (Router+Handler)│
└─────────────────┘     └─────────────────┘     └─────────────────┘
        ↑                                              ↑
        └──────────────── 或 ───────────────────────────┘
                           微信客服（公网回调 → WecomKfAdapter）
```

## 6. 合规与封号风险提醒

### 6.1 关键风险

- **NtworkWecomAdapter（企微群 hook）**：通过注入 `WXWork.exe` 获取消息，**明确违反《企业微信用户协议》**。腾讯有权利封禁账号、限制功能或追究法律责任。
- **微信客服（WecomKfAdapter）**：官方合规通道，无封号风险，但只能 1:1，不能进群。
- **MockCliAdapter**：不涉及真实通讯，无合规风险。

### 6.2 三种传输能力 / 合规 / 平台对比

| 维度 | MockCliAdapter | NtworkWecomAdapter | 微信客服（WecomKfAdapter） |
|---|---|---|---|
| **传输能力** | 本地模拟，stdout 输出 | 接管真实企微群/单聊 | 官方 1:1 客服对话 |
| **平台支持** | 任意平台（Mac/Linux/Windows） | 仅 Windows + 企微客户端 | 任意平台（需公网回调） |
| **合规性** | 不涉及真实通讯，无风险 | **违反企微协议，有封号风险** | 官方合规 |
| **群聊支持** | 模拟群聊 | 支持真实群聊 | 不支持（仅 1:1） |
| **第三方依赖** | 纯 stdlib | 需 `pip install ntwork` + 企微客户端 | 纯 stdlib |
| **实现状态** | 已实现 | Stub（待 Windows 实机） | **已实现**（公网联调待验证） |
| **适用场景** | 开发调试 / 离线测试 | 真实群私域维护（高风险） | 合规单聊客服 / 转人工 |

### 6.3 建议

- 开发阶段与离线验证**只用 Mock**；
- 真实企微 hook 仅在用户已知情并承担封号风险的前提下，于 Windows 上小规模试运行；
- 若需要**合规、稳定、可长期运行**的私域客服，**优先使用微信客服（WecomKfAdapter）**，并配合「引导用户加客服 1:1」或「群内置顶小程序入口」等运营手段转移群聊流量。
