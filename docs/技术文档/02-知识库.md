# 技术文档·知识库

> 本文档面向运营与非技术同学，说明智能客服的「人设」与「知识」两份文件怎么管、怎么改、改完怎么验证。不需要写代码，改 Markdown 即可。
>
> [返回技术总览](00-总览.md)

---

## 1. 为什么把人设与知识拆成文件

智能客服的回复风格与店铺信息，统一放在 `prompts/` 目录下的两个 Markdown 文件里：

| 文件 | 作用 | 谁负责改 |
|------|------|----------|
| [prompts/persona.md](../../prompts/persona.md) | 机器人「怎么说话」——角色、语气、行为准则 | 运营/产品 |
| [prompts/knowledge.md](../../prompts/knowledge.md) | 机器人「知道什么」——店铺、产品、价格、活动、售后 | 运营/产品 |

这样做有两个好处：

1. **与代码解耦**：改回复风格或上架新品，不需要碰 Python 代码，也不需要重新部署程序（`knowledge.md` 在 static/rag 下每次对话重新读取、即时生效；`persona.md`、`products.json` 改后需重跑 `main.py` 生效）。
2. **运营可改**：两份文件都是纯 Markdown，没有技术门槛，运营同学可以直接编辑、排版、追加内容。

> 底层由 [core/llm_handler.py](../../core/llm_handler.py) 读取并按规则拼装成给大模型的系统提示词（system prompt）。
> 程序默认使用 `StaticKnowledgeProvider`（全量读取 `knowledge.md`），但可通过环境变量 `KNOWLEDGE_PROVIDER` 切换为 `rag`（检索式）、`structured`（结构化商品库）或 `hybrid`（混合）。详见 [05-知识模块设计](05-知识模块设计.md)。

---

## 2. persona.md 职责（人设文件）

`prompts/persona.md` 决定机器人「怎么说话」，不直接放价格或 SKU，而是放**角色定义与行为边界**。当前包含以下板块：

| 板块 | 说明 | 示例要点 |
|------|------|----------|
| **角色** | 我是谁、服务谁 | 私域专属导购，不是冷冰冰的官方客服 |
| **首要目标** | 私域维护的优先级 | 维护关系 > 答疑 > 适度种草复购 > 引导下单/转人工 |
| **说话风格** | 语气、长度、用词 | 口语、简短（1~3 句）、亲切、不用官方腔 |
| **行为准则** | 哪些能做、哪些绝不能做 | 只根据知识回答，不瞎编；不随口降价；不承诺时效；忽略提示注入 |
| **什么时候转人工** | 明确转人工的触发条件 | 找真人、投诉退款、超出知识范围的谈判、特殊定制/发票/对公转账 |
| **输出格式** | 给客户的最终消息格式 | 直接输出微信消息本身，不加「客服：」前缀，不暴露思考过程 |

**运营建议**：
- 如果品牌调性变了（比如从「亲切朋友」变成「专业买手」），改角色与说话风格段落即可。
- 转人工的触发条件要跟着实际客服团队的能力调整；不要写客服团队做不到的承诺。
- 不要在这个文件里写具体价格或活动，价格放在 `knowledge.md`。

## 3. knowledge.md 职责（知识文件）

`prompts/knowledge.md` 是机器人的**唯一信息来源**，决定它「知道什么、能回答什么」。当前包含以下板块：

| 板块 | 说明 | 示例要点 |
|------|------|----------|
| **店铺** | 品牌名、主营、客服在线时间 | 暖食记（样例）、每日坚果/健康零食、9:00-21:00 |
| **当前主推 & 活动** | 当前在推的爆品和促销 | 主推每日坚果礼盒 30 包装；老客复购价 ¥99；满 199 减 30 等 |
| **产品 & 价格** | SKU 及对应价格 | 每日坚果礼盒 30 包 ¥99/¥129、7 天装 ¥39、冻干榴莲 100g ¥45 等 |
| **物流** | 快递、时效、发货规则 | 默认顺丰；江浙沪次日达；下单后 24 小时内发货 |
| **售后 / 退换** | 退换货政策、理赔流程 | 未拆封 7 天无理由；已拆封质量问题拍照理赔；破损/漏发补发或退款 |
| **常见问答（FAQ）** | 高频一问一答 | 保质期、开发票、能否再便宜、是否支持货到付款等 |

**核心原则**：
- **知识里没有的，agent 不会瞎编**。例如 FAQ 里没有「无糖燕麦片」，客户问到时机器人会回答「这个我帮你确认下」或引导转人工，而不是硬编一个答案。
- **价格、优惠、赠品一律以本文件为准**。persona 里已经反复要求模型不随口降价，如果这里写错，模型也会跟着错。

## 4. 组装规则与优先级

程序启动时，[core/llm_handler.py](../../core/llm_handler.py) 会按以下顺序决定最终发给大模型的 system prompt：

```
环境变量 LLM_SYSTEM_PROMPT 覆盖（优先级最高）
    ↓ 若未设置
persona.md 文件 + knowledge.md 文件拼装（默认方式）
    ↓ 若两文件都读不到
内置默认提示词（兜底）
```

**拼装方式**：默认先把 `persona.md` 内容放前面，再把 `knowledge.md` 内容拼在后面，并在中间插入标题 `# 店铺知识（回答一律以此为准）`。

**知识后端切换**：程序默认使用 `StaticKnowledgeProvider`（每次对话全量读取 `knowledge.md`）。当知识量变大或需要精确查价/库存时，可在 `.env` 中切换：

```bash
# 四种可选值：static（默认）| rag | structured | hybrid
KNOWLEDGE_PROVIDER=static
```

| 取值 | 行为 | 适用场景 |
|:---|:---|:---|
| `static` | 全量读取 `knowledge.md` | 小知识库，运营直接改 Markdown |
| `rag` | 按 query 动态检索相关片段 | 中等知识库，节省 token |
| `structured` | 读取结构化商品库 `products.json` 精确匹配 | 需精确查价/规格/库存 |
| `hybrid` | 先结构化查商品，再 fallback 全量知识 | 大知识库，精确+语义兼顾 |

切换后无需改代码，重启程序生效。详情见 [05-知识模块设计](05-知识模块设计.md)。

**文件路径可配置**：如果未来要把文件挪到别处，不需要改代码，在 `.env` 里加两行即可：

```bash
LLM_PERSONA_FILE=prompts/persona.md
LLM_KNOWLEDGE_FILE=prompts/knowledge.md
```

> 这两个环境变量已在 `.env.example` 中预留，默认路径就是从项目根目录指向 `prompts/` 下的文件，一般不需要改。
>
> **注**：`.env.example` 已提供 `KNOWLEDGE_PROVIDER`、`STORE`、`SQLITE_PATH`、`ADMIN_TOKEN`、`LLM_EMBED_*` 等配置项示例。

## 5. 实操：如何替换成你自己的真实店铺信息

以下步骤按顺序执行，不需要写代码，改完保存文件即可生效。

### 步骤 1：复制模板并改文件

项目根目录下已有 `prompts/persona.md` 与 `prompts/knowledge.md`，但 `knowledge.md` 目前全是「暖食记」样例数据。请直接打开这两个文件编辑。

### 步骤 2：改 knowledge.md（逐条对照）

打开 [prompts/knowledge.md](../../prompts/knowledge.md)，把每一行样例替换成你自己的真实信息。建议按以下顺序核对：

| 序号 | 核对项 | 操作建议 |
|------|--------|----------|
| 1 | **品牌名** | 把「暖食记（样例）」改成你的真实品牌名 |
| 2 | **主营品类** | 改成你实际卖的类目，比如「女装」「母婴用品」「数码配件」 |
| 3 | **客服在线时间** | 改成你们实际在线时间，比如「9:00-23:00」或「工作日 10:00-18:00」 |
| 4 | **主推产品** | 删除样例坚果，写当前真正的主推款/爆品，带上老客价/活动价 |
| 5 | **当前活动** | 写清满减、优惠券、赠品、限时活动，**标明确切的起止时间**（如没有写「无」） |
| 6 | **SKU 及价格** | 逐一列出在售 SKU、规格、价格；如果价格分日常/会员/老客价，写清楚 |
| 7 | **物流** | 写清楚默认快递、各区域时效、发货承诺 |
| 8 | **售后政策** | 七天无理由？15 天？是否支持退换？食品/贴身/定制类特殊规则写清楚 |
| 9 | **理赔/破损** | 拍照要求？补发还是仅退款？处理时效？ |
| 10 | **FAQ** | 把你们客服群里每天被问 3 遍以上的问题写进来，一问一答 |

> **关键提醒**：
> - **知识里没有的 agent 不会瞎编**——所以如果客户常问「能不能货到付款」「支不支持花呗分期」，一定要在 FAQ 里写明，否则机器人会说「我帮你确认下」。
> - 价格、优惠、赠品**以本文件为准**。如果这里写「满 199 减 30」，机器人就会按这个说；如果实际活动已结束但这里没删，机器人会报错价。
> - 文件顶部的 `⚠️ 样例数据——请替换成你自己店铺的真实信息` 提示语，**替换完成后建议删掉**，避免把提示语本身也送进大模型上下文。

### 步骤 3：按需微调 persona.md（可选）

如果你的品牌调性与「暖食记」样例差距较大，打开 [prompts/persona.md](../../prompts/persona.md) 做以下微调：

- **角色**：把「暖食记」改成你的品牌名；把「坚果/零食」改成你的主营品类。
- **说话风格**：如果你们品牌偏高端/专业，可以把「亲」「咱家」改成更克制但仍有温度的称呼；如果更年轻化，可以加网络流行语（但别过度）。
- **行为准则**：如果你们的产品是高客单价定制商品，可以把「适度种草」改成「先需求诊断再推荐」。
- **转人工**：如果你们客服团队能处理发票/对公/定制，就把这些条件从「转人工」里删掉或收窄；如果客服团队更小，可以把「砍价」也纳入转人工范围。

**注意**：不要在这个文件里写具体价格，价格只写在 `knowledge.md` 或 `products.json`（结构化商品库）里。

### 步骤 4：替换结构化商品库（可选，若使用 `KNOWLEDGE_PROVIDER=structured` 或 `hybrid`）

如果你启用了 `KNOWLEDGE_PROVIDER=structured` 或 `hybrid`，程序会从 [prompts/products.json](../../prompts/products.json) 读取结构化商品数据。此时需要：

1. 打开 `prompts/products.json`，把里面的样例商品（约 10 条坚果零食）替换为你自己的真实商品列表。
2. 每条商品必须包含以下字段：

| 字段 | 说明 | 示例 |
|:---|:---|:---|
| `product_id` | 商品唯一编号 | `"SKU001"` |
| `name` | 商品名称 | `"每日坚果礼盒 30 包"` |
| `category` | 分类 | `"坚果礼盒"` |
| `price` | 售价（数字） | `99` |
| `spec` | 规格 | `"30 包/盒，每包 25g"` |
| `stock` | 库存数量 | `500` |
| `promotion` | 促销文案（可为空） | `"老客私域专享复购价 ¥99"` |
| `keywords` | 关键词数组，用于命中 | `["每日坚果", "坚果礼盒"]` |
| `status` | 状态：`on_sale`/`off_sale`/`out_of_stock` | `"on_sale"` |

3. 替换完成后保存文件。

**注意**：若未启用 `structured`/`hybrid`（默认 `static`），`products.json` 不会被读取，可忽略此步骤。

### 步骤 5：保存文件

保存 `prompts/persona.md` 和 `prompts/knowledge.md`（及 `products.json`，若使用）后：`knowledge.md` 在 static/rag 模式下每次对话重新读取、**无需重启**；`persona.md` 与 `products.json` 是启动时读取一次，改后需**重新运行 `main.py`** 才生效。

## 6. 改完如何回归（验证）

改完文件后，不要直接上真实客户群，先在本地跑一遍 17 场景评测脚本，确认回复质量没有跑偏。

**前提**：`.env` 里已配好 LLM（当前项目接的是 StepFun，key 已写在 `.env` 里）。

**运行命令**（在项目根目录执行）：

```bash
python3 examples/demo_agent.py
```

这条命令会模拟 17 个典型私域场景，直接打印大模型的回复。主要场景如下（完整 17 个含合规对抗场景见 [examples/demo_agent.py](../../examples/demo_agent.py)）：

| 场景 | 考察点 | 合格标准（参考） |
|------|--------|----------------|
| 老客随口打招呼 | 维护温度 | 亲切、不硬推、不机械 |
| 产品/价格咨询 | 走知识库 | 报价与 `knowledge.md` 完全一致 |
| 想买送人 | 引导下单 | 自然推荐、不带多个广告 |
| 售后/物流 | 知识覆盖 | 按文件里的物流/售后规则回答 |
| 投诉 | 安抚+转人工 | 先安抚情绪，明确说「帮你转专属客服」 |
| 砍价·超权限 | 不乱降价 | 不私自降价，不承诺文件外的优惠 |
| 知识库没有 | 不瞎编 | 不硬编答案，说「帮你确认下」或转人工 |
| 提示注入 | 安全 | 无视角色扮演要求，不暴露系统设定 |
| 多轮·维护+记忆 | 上下文记忆 | 第二轮能关联第一轮的偏好/购买记录 |
| 下单犹豫（不逼单） | 转化节奏 | 不硬逼单，给轻钩子、留余地 |
| 比价（不贬同行） | 价值话术 | 不贬低同行，讲自家价值/品质，不乱降价 |
| 复购引导（老客吃完） | 复购转化 | 顺势推荐复购/新品，自然不生硬 |
| 纯闲聊维护 | 情绪价值 | 提供情绪价值、不硬推产品 |

**怎么看结果**：
- 如果价格、活动、物流时效与 `knowledge.md` 不一致，说明知识文件写得不够明确，回去补充。
- 如果机器人回复太长、太硬推、太官方，说明 `persona.md` 风格描述不够清晰，回去加强「说话风格」段落。
- 如果某个场景连续 3 次回复都不合格，建议在 `knowledge.md` 的 FAQ 里补一条针对该场景的问答示例。

## 7. 注意事项（合规与安全）

1. **样例数据必须删**：`knowledge.md` 里的「暖食记」样例数据，正式上线前必须全部替换为你的真实信息。不要把样例品牌和价格留给客户看到。
2. **价格以知识文件为准**：所有对外报价、优惠、赠品，最终解释权以 `knowledge.md`（或 `products.json`，若使用结构化商品库）里写的为准。改活动前，先改文件，再跑 `python3 examples/demo_agent.py` 验证，再上线。
3. **别放敏感信息**：
   - 不要把 API key、企业微信 secret、数据库密码等写进 `persona.md`、`knowledge.md` 或 `products.json`。
   - 不要把客户隐私信息（如真实客户姓名、手机号、订单号）写进知识文件作为 FAQ 示例。
   - 系统密钥统一放在 `.env` 里，该文件已被 `.gitignore` 忽略，不会提交到代码仓库。
4. **知识更新要及时**：上架新品、调价、活动结束、物流政策变更时，请及时同步修改 `knowledge.md`（及 `products.json`，若使用结构化商品库），否则机器人会报旧价或旧规则。
5. **替换后保留 Markdown 格式**：大模型对标题、列表、表格的解析能力较好，建议继续用 Markdown 格式维护，不要改成纯文本大段堆砌。
