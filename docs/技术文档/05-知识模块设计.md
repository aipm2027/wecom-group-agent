# 05 知识模块设计

> 定位：把「Agent 靠什么知道业务」做成可插拔组件，让 agent 在私域客服场景里既能小步快跑，又能随规模平滑升级。  
> 返回：[技术文档总览](00-总览.md)  
> **实现状态**：`StaticKnowledgeProvider`（默认）、`RagKnowledgeProvider`、`StructuredKnowledgeProvider`、`HybridKnowledgeProvider` 四者均已实现，纯 stdlib，通过环境变量 `KNOWLEDGE_PROVIDER` 在 `main.py` 中可插拔切换。

---

## 1. 问题：Agent 靠什么“知道”业务？

在电商带货的私域客服场景中，agent 必须回答商品规格、价格、库存、活动规则、售后政策等。问题是：这些知识从哪里来？

- 不能硬编码在代码里——业务知识变化快，运营需要随时改。
- 不能全靠 LLM 的预训练知识——它不知道本店今天的促销价和库存。
- 知识体量差异大：小卖家可能只有 30 个 SKU + 10 条 FAQ；大卖家可能有 500+ SKU + 大量活动规则。

因此，我们需要一个**可插拔的知识模块**，让 agent 在「小」时零依赖快速启动，在「大」时无缝接入检索和结构化存储，而**agent 核心代码不改**。

---

## 2. 三条知识路线对比

| 维度 | 纯提示词（Static） | RAG（检索增强） | 结构化商品库 |
|:---|:---|:---|:---|
| **原理** | 把知识全文塞进 system prompt | 知识切块 → embedding → 向量检索 → 取 top-k 拼进 prompt | 商品/规则建模为结构化数据，按属性/关键词精确查询 |
| **适用规模** | 几十条 SKU / 简单 FAQ | 上百 SKU / 大量 FAQ / 语义型问答 | 需要精确查价、规格、库存、促销的电商场景 |
| **优点** | 简单、零依赖、无检索误差、LLM 看到全部上下文 | 可扩展、只带相关内容、节省 token、可增量更新 | 查价/规格精确、可计算库存与促销、结果可控 |
| **缺点** | 大知识库超长/费 token、更新粗放（全量替换文件） | 需 embedding + 向量存储（可选依赖）、有检索误差、需维护索引 | 需要建模成本、对开放式语义问答弱（如“推荐一款适合送妈妈的”） |
| **成本** | 低（无额外 infra） | 中（embedding 调用 + 向量存储） | 中（建模 + 维护库） |
| **依赖** | 无（纯 Python stdlib） | embedding 接口（可复用 OpenAI 兼容接口）+ 向量存储 | 结构化存储（SQLite/JSON 即可） |
| **当前状态** | 已落地，默认启用 | 已落地，接口与实现均可用 | 已落地，接口与实现均可用 |

> **电商建议**：混合——结构化商品库负责「精确查价/规格/库存」，RAG 负责「语义型 FAQ/开放式问答」，persona 和通用指令走提示词。三者互补，不互斥。

## 3. 架构决策：可插拔 `KnowledgeProvider`

### 3.1 决策结论

延续项目中 **Adapter ↔ Router ↔ Handler** 的可插拔哲学，把知识也做成「可插拔模块」：

- **现状**：`StaticKnowledgeProvider`（读 `prompts/knowledge.md` 全量返回）与 `RagKnowledgeProvider`（知识切块 + 混合检索）均已实现；`StructuredKnowledgeProvider`（结构化商品库精确匹配）与 `HybridKnowledgeProvider`（结构化 + 语义/全量兜底）也已实现。四者纯 stdlib，零额外依赖。
- **默认**：`StaticKnowledgeProvider`，零依赖，适合当前小规模场景。
- **切换**：四者均通过 `KNOWLEDGE_PROVIDER` 环境变量在 `main.py` 中自动装配（见第 3 节及附录），Router/Handler/接口零改动。
- **组合**：`HybridKnowledgeProvider` 已实现「结构化查商品 + RAG/全量查 FAQ」的组合策略。

### 3.2 为什么这样设计？

1. **零依赖核心**：`StaticKnowledgeProvider` 纯 stdlib，不拖任何第三方，项目启动成本为零。
2. **可选依赖隔离**：RAG 所需的 embedding 接口通过 HTTP 调用（urllib），向量计算纯 Python 实现，不引入任何第三方库。
3. **平滑迁移**：知识按 `query` 现取现拼（每条消息调用一次 `retrieve(query)`），以后接入 RAG/结构化/混合 无需改 `LLMHandler._compose_system`。
4. **电商贴合**：`HybridKnowledgeProvider` 已实现「精确查商品 + 语义查 FAQ」的混合策略，可无缝切换。

## 4. 接口与现有实现

### 4.1 `KnowledgeProvider` 抽象基类

定义在 [`core/knowledge.py`](../../core/knowledge.py)：

```python
class KnowledgeProvider(ABC):
    @abstractmethod
    def retrieve(self, query: str) -> str:
        """返回与本次 query 相关的知识文本（会拼进 system prompt 的"店铺知识"部分）。

        query 是当前客户消息。静态实现可忽略它（返回全量）；检索实现据它取 top-k。
        """
        raise NotImplementedError
```

### 4.2 `StaticKnowledgeProvider`（现状默认）

```python
class StaticKnowledgeProvider(KnowledgeProvider):
    """全量返回知识文件内容（忽略 query）。当前默认，适合小知识库。"""

    def __init__(self, path: str = "prompts/knowledge.md") -> None:
        self._path = path if os.path.isabs(path) else os.path.join(_ROOT, path)

    def retrieve(self, query: str) -> str:
        try:
            with open(self._path, encoding="utf-8") as f:
                return f.read()
        except OSError:
            return ""
```

- **行为**：忽略 `query`，直接返回 `prompts/knowledge.md` 全文。
- **效果**：相当于把整份知识塞进 system prompt 的「店铺知识」区块。
- **适用**：当前项目阶段，知识量小，运营直接改 Markdown 即可。

- **原理**：读取 `prompts/products.json`（JSON 数组），按 `name` / `product_id` / `category` / `keywords` 做子串匹配。命中后返回包含价格、规格、库存、促销、状态的格式化文本。

    """混合知识：先结构化查商品，再 fallback 语义/全量查 FAQ。"""

- **原理**：先执行 `primary`（默认 `StructuredKnowledgeProvider`）查商品；若未命中或出错，再执行 `fallback`（默认 `StaticKnowledgeProvider`）查全量知识 / FAQ。两者结果拼接后返回。

### 4.6 `LLMHandler` 中的三级优先级组装

定义在 [`core/llm_handler.py`](../../core/llm_handler.py)，关键逻辑：

```python
def _compose_system(self, query: str) -> str:
    """组装 system prompt：LLM_SYSTEM_PROMPT 覆盖 > persona + 知识模块检索结果 > 内置默认。

    知识按 query 从 KnowledgeProvider 取（静态实现返回全量、检索实现返回相关片段），
    所以每条消息现取现拼——这样以后换 RAG 无需改这里。
    """
    if self._override:
        return self._override
    base = self._persona.strip() if self._persona else self._DEFAULT_SYSTEM
    parts = [base]
    knowledge = self._knowledge.retrieve(query).strip()
    if knowledge:
        parts.append("# 店铺知识（回答一律以此为准）\n\n" + knowledge)
    return "\n\n".join(parts)
```

**优先级（从高到低）**：

1. **`LLM_SYSTEM_PROMPT` 环境变量覆盖**：若设置，直接作为完整 system prompt，忽略 persona 和知识模块。
2. **`persona`（`prompts/persona.md`）+ 知识模块 `retrieve(query)`**：默认路径。知识模块返回的文本被包裹在 `# 店铺知识` 标题下注入。
3. **内置默认 `_DEFAULT_SYSTEM`**：兜底，确保即使文件缺失也能运行。

**注入时机**：`reply()` 中为每条用户消息调用 `_compose_system(msg.content)`，即**现取现拼**。这意味着：
- 换 `RagKnowledgeProvider` 后，不需要修改 `LLMHandler` 的任何代码；
- RAG 可以按 query 做动态检索，返回最相关的片段而非全文。

## 5. RAG 实现细节与优化

> 以下描述为 `RagKnowledgeProvider` 的**已实现细节**，源码位于 `core/knowledge.py`，测试见 `tests/test_rag_offline.py`。

### 5.1 逻辑切块

不按固定字数硬切（避免切断价格/规格），而是按 Markdown 语义结构切：

- **切分依据**：ATX 标题（`# `）为章节，每个列表项（`- `）独立成块，段落正文独立成块。
- **上下文保留**：每个 chunk 带所属小节标题，例如 `# 当前主推 & 活动\n- 【主推】每日坚果礼盒...`，确保 LLM 知道该条目的所属分类。
- **样例数据过滤**：跳过含「样例数据」的行，避免把占位示例注入 prompt。

### 5.2 混合检索（三路取并集、按原文顺序输出）

检索不是单一向量 top-k，而是三路取并集后按原文索引排序，避免语义模型遗漏关键信息：

| 通路 | 机制 | 说明 |
|:---|:---|:---|
| **① 语义** | embedding 余弦 top-k（默认 `RAG_TOP_K=4`） | query 与 chunk 做 L2 归一化后点积，取最相似 k 个。 |
| **② 关键词** | query 子串命中 + 字符 bigram 交集≥2 | 提取 query 中长度≥2 的 token 做子串匹配；未命中时再用字符 2-gram 交集≥2 兜底，中文友好。 |
| **③ 常驻关键块** | 无条件纳入 | chunk 标题或内容含「活动/主推/优惠/促销/满减/折扣/特价/限时」等关键词，无条件加入结果。 |

**输出规则**：三路取并集后，按 chunk 在原文中的原始索引升序排列，不按相似度重排——这样保持知识原有的结构顺序（如活动在前、商品在后），LLM 阅读更自然。

### 5.3 "永不比全量塞更差"的多重兜底

RAG 宁可多给也不漏答，设置多重底线确保检索结果不会比全量塞更差：

| 兜底场景 | 行为 |
|:---|:---|
| **小库直通** | chunk 数 ≤ `RAG_SMALL_KB_MAX`（默认 15）时直接返回全文，不 embedding、不检索。小库用 RAG 无益。 |
| **三路结果为空** | 若语义/关键词/常驻块均无命中 → 直接返回全文。 |
| **疑似漏召** | 检索结果与 query 无字符 bigram 重叠（说明可能没召回真正被问的内容）→ 回退全文。 |
| **任何异常** | embedding 失败、无 key、网络错误、解析失败等 → stderr 打印错误（过滤 api_key）→ 回退全文。 |

### 5.4 纯 stdlib + 可注入 embedder

- **无 numpy/第三方**：向量化走 HTTP（urllib，OpenAI 兼容 `/embeddings`）或注入 `embed_fn`；余弦相似度纯 Python（L2 归一化 + 点积）。
- **可注入 embedder**：构造时传入 `embed_fn=callable`，即可绕过 HTTP，便于离线测试、接入私有化 embedding 或本地模型。
- **HTTP 路径**：默认复用 `LLM_BASE_URL`（或单独 `LLM_EMBED_URL`），调用兼容 `/embeddings` 接口，支持批量请求。

### 5.5 索引缓存（内存 + 磁盘）

- **内存缓存**：按知识内容 SHA256 做键。内容不变时，同进程内二次检索直接复用索引，只重新 embed query。
- **磁盘缓存**：HTTP 路径下，索引持久化到 `data/rag_cache/{hash}-{model}.json`（`.gitignore` 已排除 `data/`）。
- **缓存键含模型名**：文件名包含 embedding 模型名，换模型自动命中不同缓存，避免加载旧维度向量导致静默截断。
- **注入 embed_fn 时**：不走磁盘缓存，仅内存缓存，保证隔离性。
- **自动重建**：知识内容变化（SHA256 变）→ 自动重新 embed chunks 并重建缓存。

### 5.6 配置项

全部可通过环境变量或构造参数覆盖：

| 配置项 | 环境变量 | 默认值 | 说明 |
|:---|:---|:---|:---|
| embedding URL | `LLM_EMBED_URL` | 取 `LLM_BASE_URL` | OpenAI 兼容 embedding 端点。 |
| embedding 模型 | `LLM_EMBED_MODEL` | `text-embedding-3-small` | 实际部署时改成供应商 embedding 模型。 |
| API key | `LLM_API_KEY` | — | 复用 LLM 的 key。 |
| 语义 top-k | `RAG_TOP_K` | 4 | 语义通路取前 k 个 chunk。 |
| 小库阈值 | `RAG_SMALL_KB_MAX` | 15 | chunk 数 ≤ 此值时直通全文。 |
| 超时 | `RAG_TIMEOUT` | 30 秒 | embedding HTTP 调用超时。 |

### 5.7 如何切换

- **`KNOWLEDGE_PROVIDER` 环境变量**：在 `main.py` 的 `build_knowledge()` 中自动装配，无需改 `LLMHandler` 代码：

```python
# main.py
from core.knowledge import RagKnowledgeProvider
from core.llm_handler import LLMHandler

handler = LLMHandler(knowledge=build_knowledge())  # 自动按 KNOWLEDGE_PROVIDER 装配
```

| `KNOWLEDGE_PROVIDER` | 实际装配的 Provider | 说明 |
|:---|:---|:---|
| `static`（默认） | `StaticKnowledgeProvider` | 全量塞 knowledge.md |
| `rag` | `RagKnowledgeProvider()` | 混合检索，动态取片段 |
| `structured` | `StructuredKnowledgeProvider()` | 结构化商品库精确查价 |
| `hybrid` | `HybridKnowledgeProvider(StructuredKnowledgeProvider(), StaticKnowledgeProvider())` | 结构化查商品 + 全量 FAQ 兜底 |

> 实现细节：`static` 时 `build_knowledge()` 返回 `None`，由 `LLMHandler` 构造时自行实例化默认的 `StaticKnowledgeProvider`（该路径会读取 `LLM_KNOWLEDGE_FILE`）；`rag`/`structured`/`hybrid` 三种由 `build_knowledge()` 显式装配后经 `knowledge=` 注入。

### 5.8 诚实边界与已知限制

- **小库无益**：已用「小库直通」兜底，chunk 数少时直接返回全文，等价 Static。
- **精确查价/规格**：RAG 做语义召回，对精确价格/规格查询不如结构化商品库可靠。建议用结构化商品库（`KNOWLEDGE_PROVIDER=structured` 或 `hybrid`）负责「价/规格/库存」，RAG 负责语义 FAQ。
- **概览类问题**：如「你们卖什么」，RAG 可能只召回部分商品而非全部——由 "never-worse" 兜底（若结果与 query 无 bigram 重叠则回退全文）守住底线。
- **embedding 端点依赖**：当前 StepFun 那把 key 未见 embedding 模型，需另配一个 OpenAI 兼容 embedding 端点；未配置则 RAG 自动回退全量（等价 Static，不影响可用）。

### 5.9 离线测试

`tests/test_rag_offline.py` 包含 8 项测试，注入确定性假 embedder，纯离线、无网络、无真实 key：

1. 小库直通（不调用 embed_fn）
2. 语义检索（命中目标、排除无关项）
3. 关键词兜底（语义失效时子串/bigram 仍召回）
4. 常驻块无条件纳入
5. 原序排列（按 chunk 原始索引，非相似度重排）
6. 出错回退（异常时降级全量，stderr 不泄漏 key）
7. 无 key 回退（未配置 api_key 时返回全文）
8. 缓存重建（内容变更后自动重建索引）

## 6. 结构化商品库（已落地）与混合策略

### 6.1 为什么还需要结构化商品库？

RAG 擅长「语义匹配」，但电商场景中有大量**精确查询**：

- 「商品 A 多少钱？」——需要精确对应 SKU，不能含糊。
- 「有没有 200g 装的？」——需要属性匹配（重量=200g）。
- 「库存够吗？」——需要结构化字段计算。
- 「满 199 减 30 的活动包含这个吗？」——需要规则引擎或结构化查询。

RAG 对这类问题可能「检索到相近但不准确」的信息，导致幻觉或错误报价。

### 6.2 结构化商品库（已落地）

- **数据源**：`prompts/products.json`（JSON 数组，样例约 10 条坚果零食）。
- **模型**：每条商品包含 `product_id`、`name`、`category`、`price`、`spec`、`stock`、`promotion`、`keywords`、`status`。
- **查询方式**：`name` / `product_id` / `category` / `keywords` 子串精确匹配，非向量相似度。
- **返回格式**：命中商品按「名称 / 价格 / 规格 / 库存 / 促销 / 状态」拼接为结构化文本，注入 system prompt。
- **兜底**：文件缺失 / JSON 损坏 / 未命中均返回空串，不抛异常；`out_of_stock` 标注「已售罄」，`off_sale` 标注「已下架」。

### 6.3 混合策略（已落地）

`HybridKnowledgeProvider` 已在 `core/knowledge.py` 实现：

```
用户 query
    ↓
[结构化商品库查询] —— 命中商品？
    ↓ 命中                    ↓ 未命中 / 异常
  精确结果（价格/规格/库存）   fallback 全量/语义知识
    ↓                              ↓
  合并注入 system prompt → LLM 生成回复
```

- **默认装配**：`main.py` 中 `KNOWLEDGE_PROVIDER=hybrid` 时自动装配为 `HybridKnowledgeProvider(StructuredKnowledgeProvider(), StaticKnowledgeProvider())`。
- **行为**：结构化优先，fallback 兜底，各自 try/except 隔离，任一环节失败不影响整体可用。

## 7. 选型与迁移路径

| 阶段 | 知识体量 | 推荐方案 | 操作 |
|:---|:---|:---|:---|
| **起步** | < 50 条 SKU / 简单 FAQ | `StaticKnowledgeProvider` | 运营改 `prompts/knowledge.md`，零代码、零依赖。 |
| **成长** | 50~200 条 SKU / FAQ 增多 | `RagKnowledgeProvider` | `KNOWLEDGE_PROVIDER=rag`，一行切换，agent 代码不改。 |
| **规模化** | 200+ SKU / 多活动 / 需精确查库存价格 | `HybridKnowledgeProvider` | `KNOWLEDGE_PROVIDER=hybrid`，自动组合结构化 + 全量兜底；`RAG` 仍可负责非结构化 FAQ。 |

**核心原则**：
- **小走 Static**：现在就用，已经落地。
- **Static → Rag/Structured/Hybrid 换一行**：`KNOWLEDGE_PROVIDER=rag/structured/hybrid` 即可切换，因为 `LLMHandler` 只依赖 `KnowledgeProvider.retrieve(query)` 接口。
- **逐步叠加**：不推翻重来，Hybrid 是「新增」而非「替换」。

## 8. 真实数据接入

最后阶段，需要把「占位知识」替换为真实业务数据：

### 8.1 替换 `knowledge.md`（现状即可操作）

- 当前运营可直接编辑 `prompts/knowledge.md`，内容会被 `StaticKnowledgeProvider` 全量读取，或被 `RagKnowledgeProvider` 按语义切块后检索。
- 建议格式：Markdown 分级标题，方便运营阅读和 RAG 自动切块。

示例结构：

```markdown
# 店铺知识

## 商品列表
- 商品A：XX 规格，售价 99 元，适合...
- 商品B：XX 规格，售价 199 元，...

## 售后规则
- 7 天无理由退换（未拆封）
- 质量问题包运费

## 当前活动
- 满 199 减 30
```

### 8.2 接入商品库（现状即可操作）

- 编辑 `prompts/products.json`（JSON 数组），样例约 10 条坚果零食。字段说明：

| 字段 | 类型 | 说明 |
|:---|:---|:---|
| `product_id` | string | 商品唯一编号 |
| `name` | string | 商品名称，匹配 query 子串 |
| `category` | string | 类目，匹配 query 子串 |
| `price` | number | 售价，格式化后注入 prompt |
| `spec` | string | 规格，格式化后注入 prompt |
| `stock` | number | 库存数，格式化后注入 prompt |
| `promotion` | string | 促销文案，非空时追加 |
| `keywords` | list[string] | 匹配关键词数组，命中即召回 |
| `status` | string | `on_sale` / `off_sale` / `out_of_stock` |

- 替换后设置 `KNOWLEDGE_PROVIDER=structured` 或 `hybrid` 即可启用。不追求实时同步，以「手动替换 JSON + 重启」为主。

### 8.3 与人工接管的协同

- 当知识库无法回答（RAG 无命中、结构化查询无结果）时，agent 目前返回「转人工」话术。
- 已落地：LLM 输出 `[[转人工]]` → `LLMHandler` 剥离并 `session.mark_needs_human` → `Router` 触发 `on_escalate` 回调；`session.human_controlled=True` 时 Router 静默。
- **规划**：升级为结构化工单系统 + 人工工作台前端（见 [需求总览](../需求文档/00-总览.md) 中「人工接管」章节）。
