# 05 知识模块设计

> 定位：把「Agent 靠什么知道业务」做成可插拔组件，让 agent 在私域客服场景里既能小步快跑，又能随规模平滑升级。  
> 返回：[技术文档总览](00-总览.md)

---

## 1. 问题：Agent 靠什么“知道”业务？

在电商带货的私域客服场景中，agent 必须回答商品规格、价格、库存、活动规则、售后政策等。问题是：这些知识从哪里来？

- 不能硬编码在代码里——业务知识变化快，运营需要随时改。
- 不能全靠 LLM 的预训练知识——它不知道本店今天的促销价和库存。
- 知识体量差异大：小卖家可能只有 30 个 SKU + 10 条 FAQ；大卖家可能有 500+ SKU + 大量活动规则。

因此，我们需要一个**可插拔的知识模块**，让 agent 在「小」时零依赖快速启动，在「大」时无缝接入检索和结构化存储，而**agent 核心代码不改**。

---

## 2. 三条知识路线对比

| 维度 | 纯提示词（Static） | RAG（检索增强） | 结构化商品库 |
|:---|:---|:---|:---|
| **原理** | 把知识全文塞进 system prompt | 知识切块 → embedding → 向量检索 → 取 top-k 拼进 prompt | 商品/规则建模为结构化数据，按属性/关键词精确查询 |
| **适用规模** | 几十条 SKU / 简单 FAQ | 上百 SKU / 大量 FAQ / 语义型问答 | 需要精确查价、规格、库存、促销的电商场景 |
| **优点** | 简单、零依赖、无检索误差、LLM 看到全部上下文 | 可扩展、只带相关内容、节省 token、可增量更新 | 查价/规格精确、可计算库存与促销、结果可控 |
| **缺点** | 大知识库超长/费 token、更新粗放（全量替换文件） | 需 embedding + 向量存储（可选依赖）、有检索误差、需维护索引 | 需要建模成本、对开放式语义问答弱（如“推荐一款适合送妈妈的”） |
| **成本** | 低（无额外 infra） | 中（embedding 调用 + 向量存储） | 中（建模 + 维护库） |
| **依赖** | 无（纯 Python stdlib） | embedding 接口（可复用 StepFun/OpenAI 兼容接口）+ 向量存储 | 结构化存储（SQLite/JSON 即可） |
| **当前状态** | ✅ **已落地**，默认启用 | 📝 **规划**，接口已预留 stub | 📝 **规划**，未实现 |

> **电商建议**：混合——结构化商品库负责「精确查价/规格/库存」，RAG 负责「语义型 FAQ/开放式问答」，persona 和通用指令走提示词。三者互补，不互斥。

## 3. 架构决策：可插拔 `KnowledgeProvider`

### 3.1 决策结论

延续项目中 **Adapter ↔ Router ↔ Handler** 的可插拔哲学，把知识也做成「可插拔模块」：

- **现状**：使用 `StaticKnowledgeProvider`（读 `prompts/knowledge.md` 全量返回），零依赖，适合当前小规模场景。
- **预留**：`RagKnowledgeProvider` 接口已定义，但抛出 `NotImplementedError`，规模化时直接实现即可。
- **注入点**：`LLMHandler` 构造时接受 `knowledge=Provider`；默认自动创建 `StaticKnowledgeProvider`。换 RAG 时只需改初始化一行，agent 其余代码不动。

### 3.2 为什么这样设计？

1. **零依赖核心**：`StaticKnowledgeProvider` 纯 stdlib，不拖任何第三方，项目启动成本为零。
2. **可选依赖隔离**：RAG 所需的 embedding 库、向量库均通过 guard import 使用，不污染核心。
3. **平滑迁移**：知识按 `query` 现取现拼（每条消息调用一次 `retrieve(query)`），以后接入 RAG 无需改 `LLMHandler._compose_system`。
4. **电商贴合**：预留结构化商品库扩展点，未来可与 RAG 组合，实现「精确查商品 + 语义查 FAQ」的混合策略。

## 4. 接口与现有实现

### 4.1 `KnowledgeProvider` 抽象基类

定义在 [`core/knowledge.py`](../../core/knowledge.py)：

```python
class KnowledgeProvider(ABC):
    @abstractmethod
    def retrieve(self, query: str) -> str:
        """返回与本次 query 相关的知识文本（会拼进 system prompt 的"店铺知识"部分）。

        query 是当前客户消息。静态实现可忽略它（返回全量）；检索实现据它取 top-k。
        """
        raise NotImplementedError
```

### 4.2 `StaticKnowledgeProvider`（现状）

```python
class StaticKnowledgeProvider(KnowledgeProvider):
    """全量返回知识文件内容（忽略 query）。当前默认，适合小知识库。"""

    def __init__(self, path: str = "prompts/knowledge.md") -> None:
        self._path = path if os.path.isabs(path) else os.path.join(_ROOT, path)

    def retrieve(self, query: str) -> str:
        try:
            with open(self._path, encoding="utf-8") as f:
                return f.read()
        except OSError:
            return ""
```

- **行为**：忽略 `query`，直接返回 `prompts/knowledge.md` 全文。
- **效果**：相当于把整份知识塞进 system prompt 的「店铺知识」区块。
- **适用**：当前项目阶段，知识量小，运营直接改 Markdown 即可。

### 4.3 `RagKnowledgeProvider`（规划 stub）

```python
class RagKnowledgeProvider(KnowledgeProvider):
    """检索式知识（stub，规模化再实现）。"""

    def retrieve(self, query: str) -> str:
        raise NotImplementedError(
            "RagKnowledgeProvider 尚未实现（规模化再上）；当前请用 StaticKnowledgeProvider。"
        )
```

- **状态**：📝 **规划**，接口已占位，实现时机见第 5 章。

### 4.4 `LLMHandler` 中的三级优先级组装

定义在 [`core/llm_handler.py`](../../core/llm_handler.py)，关键逻辑：

```python
def _compose_system(self, query: str) -> str:
    """组装 system prompt：LLM_SYSTEM_PROMPT 覆盖 > persona + 知识模块检索结果 > 内置默认。

    知识按 query 从 KnowledgeProvider 取（静态实现返回全量、检索实现返回相关片段），
    所以每条消息现取现拼——这样以后换 RAG 无需改这里。
    """
    if self._override:
        return self._override
    base = self._persona.strip() if self._persona else self._DEFAULT_SYSTEM
    parts = [base]
    knowledge = self._knowledge.retrieve(query).strip()
    if knowledge:
        parts.append("# 店铺知识（回答一律以此为准）\n\n" + knowledge)
    return "\n\n".join(parts)
```

**优先级（从高到低）**：

1. **`LLM_SYSTEM_PROMPT` 环境变量覆盖**：若设置，直接作为完整 system prompt，忽略 persona 和知识模块。
2. **`persona`（`prompts/persona.md`）+ 知识模块 `retrieve(query)`**：默认路径。知识模块返回的文本被包裹在 `# 店铺知识` 标题下注入。
3. **内置默认 `_DEFAULT_SYSTEM`**：兜底，确保即使文件缺失也能运行。

**注入时机**：`reply()` 中为每条用户消息调用 `_compose_system(msg.content)`，即**现取现拼**。这意味着：
- 换 `RagKnowledgeProvider` 后，不需要修改 `LLMHandler` 的任何代码；
- RAG 可以按 query 做动态检索，返回最相关的片段而非全文。

## 5. RAG 落地方案（规划）

> ⚠️ 以下方案为**规划阶段**，尚未实现。接口已预留（`RagKnowledgeProvider`），实现时直接填充即可。

### 5.1 数据流

```
原始知识源（商品表/FAQ/活动规则）
      ↓
  文本切块（按条目/段落）
      ↓
  Embedding（调用 StepFun/OpenAI 兼容接口）
      ↓
  向量存储（[向量, 原文, 元数据]）
      ↓
  查询期：query → embedding → 余弦 top-k → 拼片段 → 注入 system prompt
```

### 5.2 各层要点

| 环节 | 现状/方案 | 说明 |
|:---|:---|:---|
| **数据源** | 📝 规划接入 | 商品描述、FAQ、售后规则、活动文案。建议保留原始 Markdown/JSON 作为「源文件」，索引从源文件重建。 |
| **切块策略** | 📝 规划 | 按语义条目切（一个商品一条、一个 QA 一对）。避免切太碎导致丢失上下文，也避免切太大导致检索噪声。 |
| **Embedding** | 📝 复用 StepFun | 项目当前 LLM 使用 StepFun（OpenAI 兼容）。embedding 可直接复用其接口，无需额外供应商。作为可选依赖 guard import。 |
| **向量存储** | 📝 两档方案 | **小规模**：内存 dict + JSON 持久化（零额外依赖）；**大规模**：接入向量库（如 faiss、chromadb），同样 guard import。 |
| **检索** | 📝 余弦 top-k | query embedding 后与库内向量做余弦相似度，取 top-k（如 k=3~5），把原文片段按相关性排序拼接。 |
| **注入** | 📝 已有接口 | 直接返回拼接后的文本片段，`LLMHandler._compose_system` 会自动把它包进 `# 店铺知识` 区块。 |
| **更新索引** | 📝 规划 | 提供显式重建接口（如 `rebuild_index()`），支持热更新；不追求实时秒级同步，运营改完后主动触发重建即可。 |
| **防幻觉/引用** | 📝 规划 | 在返回片段中保留来源标识（如 `【商品A】`），并在 prompt 里加指令「如果知识库中没有，请引导联系人工，不要编造」。 |

### 5.3 实现位置

- 代码位置：`core/knowledge.py` 中的 `RagKnowledgeProvider`。
- 索引数据目录：建议放在 `data/rag_index/`（.gitignore），避免把索引文件提交进仓库。
- 可选依赖管理：embedding 客户端和向量库在文件顶部做 `try/except ImportError`，未安装时抛明确错误提示，不影响 `StaticKnowledgeProvider` 使用。

## 6. 结构化商品库（可选）与混合策略

### 6.1 为什么还需要结构化商品库？

RAG 擅长「语义匹配」，但电商场景中有大量**精确查询**：

- 「商品 A 多少钱？」——需要精确对应 SKU，不能含糊。
- 「有没有 200g 装的？」——需要属性匹配（重量=200g）。
- 「库存够吗？」——需要结构化字段计算。
- 「满 199 减 30 的活动包含这个吗？」——需要规则引擎或结构化查询。

RAG 对这类问题可能「检索到相近但不准确」的信息，导致幻觉或错误报价。

### 6.2 结构化商品库（规划）

- **存储**：SQLite（项目已规划持久化）或 JSON。
- **模型**：商品表（SKU、名称、价格、规格、库存、分类、状态）+ 活动规则表 + 库存变动表。
- **查询方式**：关键词/属性精确查询，不是向量相似度。
- **与 RAG 的关系**：并列，不是替代。agent 可以先尝试结构化查商品，若未命中再 fallback 到 RAG 查语义 FAQ。

### 6.3 混合策略（推荐）

```
用户 query
    ↓
[意图识别] —— 是否涉及精确商品价格/规格/库存/活动？
    ↓ 是                          ↓ 否
  结构化商品库查询                RAG 语义检索
    ↓                              ↓
  精确结果（如价格 99 元）      相关 FAQ/知识片段
    ↓                              ↓
  合并注入 system prompt → LLM 生成回复
```

> **说明**：意图识别可以先做简单规则匹配（含「多少钱」「库存」「规格」等关键词），未来再考虑用轻量模型分类。当前阶段，混合策略是**规划**，不急于实现。

## 7. 选型与迁移路径

| 阶段 | 知识体量 | 推荐方案 | 操作 |
|:---|:---|:---|:---|
| **起步** | < 50 条 SKU / 简单 FAQ | `StaticKnowledgeProvider` | 运营改 `prompts/knowledge.md`，零代码、零依赖。 |
| **成长** | 50~200 条 SKU / FAQ 增多 | `StaticKnowledgeProvider`（仍可用）或切换 `RagKnowledgeProvider` | 实现 RAG，把 `knowledge` 参数换成 `RagKnowledgeProvider(...)`，agent 代码不改。 |
| **规模化** | 200+ SKU / 多活动 / 需精确查库存价格 | **混合**：结构化商品库 + RAG FAQ | 新增结构化商品库，LLMHandler 中按意图分流；`KnowledgeProvider` 本身可保持为 RAG 负责非结构化知识。 |

**核心原则**：
- **小走 Static**：现在就用，已经落地。
- **变大换 Provider 不改 agent**：因为 `LLMHandler` 只依赖 `KnowledgeProvider.retrieve(query)` 接口，换后端只需改初始化时的 `knowledge=` 参数。
- **逐步叠加**：不推翻重来，结构化商品库是「新增」而非「替换」。

## 8. 真实数据接入

最后阶段，需要把「占位知识」替换为真实业务数据：

### 8.1 替换 `knowledge.md`（现状即可操作）

- 当前运营可直接编辑 `prompts/knowledge.md`，内容会被 `StaticKnowledgeProvider` 全量读取。
- 建议格式：Markdown 分级标题，方便运营阅读和以后自动切块。

示例结构：

```markdown
# 店铺知识

## 商品列表
- 商品A：XX 规格，售价 99 元，适合...
- 商品B：XX 规格，售价 199 元，...

## 售后规则
- 7 天无理由退换（未拆封）
- 质量问题包运费

## 当前活动
- 满 199 减 30
```

### 8.2 接入商品库导入（规划）

- 未来提供脚本/接口，把电商后台导出的商品表（CSV/JSON）导入到结构化商品库 + RAG 索引。
- 导入后触发一次索引重建，确保向量库与源数据同步。
- 不追求实时同步，以「批次导入 + 手动重建」为主，降低复杂度。

### 8.3 与人工接管的协同

- 当知识库无法回答（RAG 无命中、结构化查询无结果）时，agent 目前返回「转人工」话术。
- **规划**：升级为结构化 `escalation` 信号（见 [需求总览](../需求文档/00-总览.md) 中「人工接管」章节），触发后端工单 → 工作台接管，而非仅文字告知用户。
