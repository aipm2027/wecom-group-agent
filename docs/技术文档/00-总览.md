## 1. 技术定位

`wecom-group-agent` 是一个电商带货「私域维护」智能客服 agent 的骨架实现。核心定位是**接管微信群实时对话并智能回复**，以“懂行的贴心导购”角色维护客户关系、答疑、适度种草促复购，并在必要时引导下单或转人工。整个系统采用**传输无关的三层架构**（传输适配器 ↔ 路由核心 ↔ 回复大脑/会话上下文），全部基于 Python 标准库实现，零第三方依赖，任何环境可直接 `python3 main.py` 运行。

## 4. 一条消息的完整数据流

以下以 **mock 交互模式** 为例，展示一条消息从“收到”到“发出”再到“记忆”的完整时序（单聊或群聊 `@bot` 触发场景）。

**T0 — 用户输入**
```
群A|张三|@bot 这个每日坚果礼盒怎么卖
```

**T1 — 适配器收到并规范化**
`MockCliAdapter._run_interactive()` 从 `stdin` 读取，按 `|` 切分，识别内容含 `@bot`，设置 `is_at_bot=True`，剥掉 `@bot` 后组装为 `Message` 对象：

```python
Message(
    chat_id="群A", chat_type="group", msg_id="cli-1",
    sender_id="张三", sender_name="张三",
    content="这个每日坚果礼盒怎么卖",
    is_at_bot=True, raw={"line": "群A|张三|@bot 这个每日坚果礼盒怎么卖"}
)
```

**T2 — 交给 Router.on_message**
`adapter.start()` 的回调即 `router.on_message(msg)`，进入 `Router` 处理流水线：

| 步骤 | 动作 | 说明 |
|------|------|------|
| 2.1 去重 | `msg_id="cli-1"` 不在 `OrderedDict` 中，通过；若重复则直接丢弃。 | 上限 `DEDUP_MAX=5000`，超限时淘汰最旧。 |
| 2.2 记入会话 | `session = sessions.get("群A"); session.add(msg)` | 无论是否回复，都保留历史。 |
| 2.3 触发判断 | `should_respond()`：群聊且 `is_at_bot=True` → **回**；非 text 不回。 | 单聊全回；群里没 `@bot` 直接静默。 |
| 2.4 限流 | 检查 `_last_reply_at.get("群A")`，距上次回复是否 `< 2.0s`。 | 若限流则静默丢弃，不触发 handler。 |
| 2.5 生成回复 | `reply = handler.reply(msg, session)` | 当前 `HANDLER=llm` 时走 `LLMHandler`。 |
| 2.6 回发 | `adapter.send("群A", reply)` | 先发成功，再更新限流时间戳。 |
| 2.7 记忆 bot 回复 | `session.add(Message(..., sender_id=BOT_SENDER_ID, content=reply))` | 把机器人自己说的话以 `__bot__` 角色写回上下文，供多轮记忆。 |

**T3 — 发送成功**
`MockCliAdapter.send()` 打印到 `stdout`：

```
[发送→群A] 亲，每日坚果礼盒 30 包装老客复购价 ¥99 哦，现在下单还能参加满减活动～
```

**T4 — 多轮上下文生效**
若同一用户继续问“那我要送人推荐哪个”，`LLMHandler._build_messages()` 会从 `session.history` 取最近 `LLM_MAX_HISTORY`（默认 10）条，其中 bot 的上一条回复映射为 `role=assistant`，用户新消息映射为 `role=user` 并带 `昵称：`前缀，形成完整多轮上下文送入 LLM。

## 5. 技术选型与理由

| 选型 | 理由 |
|------|------|
| **纯 Python 标准库，零第三方依赖** | 降低部署摩擦（任意装 python3 的环境直接跑）、减少供应链攻击面、避免依赖版本冲突。LLM 调用用 `urllib.request`，JSON 解析用 `json`，数据结构用 `collections.deque/OrderedDict/dataclasses`。 |
| **可插拔 Adapter / 可插拔 Handler** | `Adapter` 抽象让“换传输”成为纯配置问题：mock 验证、`ntwork` 接 Windows 企微 hook、未来 `kf` 接微信客服官方 API，均只新增一个文件，**核心 Router 一行不改**。`Handler` 抽象让“换大脑”同样零侵入：占位 `EchoHandler` 验证链路，替换为 `LLMHandler` 即可接入大模型，以后新增规则引擎/微调模型同理。 |
| **OpenAI 兼容接口 + 纯 urllib** | 当前接 **StepFun（阶跃星辰）**，其 API 为 OpenAI 兼容格式；以后切 DeepSeek、自托管 vLLM 等，只需改 `LLM_BASE_URL`/`LLM_MODEL` 环境变量，**不引入任何供应商 SDK**。 |
| **环境变量驱动配置** | `.env` 文件（被 `.gitignore` 忽略）+ `os.environ` 读取，无配置框架依赖；`main.py` 中 `load_env_file()` 不覆盖已有环境变量，方便 CI/CD 覆盖。 |
| **transport 注入可测性** | `LLMHandler.__init__(transport=...)` 可短路真实网络，离线验证消息组装、角色映射、截断、降级逻辑，无需 mock server 或伪造 API key。 |

## 6. 硬约束

| 约束 | 说明 | 当前状态 |
|------|------|----------|
| **微信群接管仅 Windows hook** | 微信群实时消息官方无 API，只能 hook 注入企微客户端（`WXWork.exe`）。这违反企微用户协议，有封号风险。 | `NtworkWecomAdapter` 为 stub，待 Windows 环境实现。 |
| **Mac 无法注入企微 hook** | Mac 因 SIP（系统完整性保护）+ 无成熟逆向工具，无法对企微客户端做同等注入。 | 本地开发/验证必须使用 `MockCliAdapter`（`MOCK=1`）。 |
| **微信客服仅支持 1:1 单聊** | 微信客服 API 是官方合规接口，但只能做一对一客服，**进不了群**。 | 微信客服适配器（`ADAPTER=kf`）待接入。 |
| **当前运行环境为 Apple Silicon M5 Mac** | 真实企微 hook 无法在本机运行。 | 本阶段目标：mock 在 Mac 跑通全链路；真实 ntwork 与微信客服待后续环境。 |
| **API key 合规** | `LLM_API_KEY` 存放在 `.env`（已 `.gitignore`），切勿提交到仓库；建议定期轮换。 | 当前 `.env` 使用 StepFun key。 |
