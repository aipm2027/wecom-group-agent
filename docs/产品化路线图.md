# 产品化路线图

> 产出方式:三视角(PM/算法/工程)并行审计 → 每条发现独立对抗验证(33 agents,24 条原始发现,8 条存活、16 条被驳回)→ 综合。2026-07-10,基于 HEAD `9abea4a`。
> 质量基线(2026-07-10):检索层离线评测 **17/17**(可回归);回复层在线评测 **21/21**(StepFun step-3.7-flash + hybrid,含 4 个合规红线案例)。

## 产品定位

面向 **1-2 人运营 + 兼职技术** 的私域电商小团队,提供零第三方依赖、纯 Python 标准库的微信客服 AI Agent;解决 7x24 自动回复、知识库驱动、转人工闭环问题;凭"合规微信客服通道 + 已验证的回复质量基线 + 一键容器部署"敢用。

## MVP 形态判断(已共识,DISCUSSION.md #1/#2)

**微信客服官方 1:1 通道(WecomKfAdapter)= MVP 唯一合规真渠道**;ntwork 企微群 hook 降为 P2 远期(仅 Windows + 违规封号风险)。官方通道无封号风险、AES-256 回调加解密已就绪、可挂视频号/公众号/小程序客服入口——私域电商在腾讯生态内不损失场景覆盖。

## P0 — 不做就落不了地

### 1. .env 配置启动自检(Owner: CONV-B,S)
- **做什么**:`main.py`/`api_server.py` 入口增加配置校验:关键项缺失/组合非法时打印人可读错误(含修复指引)并以非 0 退出,不进入任何静默降级。
- **为什么**(对抗验证确认的三个静默深坑):
  - 漏配 `LLM_API_KEY` → LLMHandler 静默降级为固定兜底话术(`core/llm_handler.py`),运营者花着钱以为 AI 在工作;
  - Mac 上忘配 `ADAPTER=kf`/`MOCK=1` → 默认走 ntwork stub 直接 `NotImplementedError`,非技术用户不可理解;
  - `ADAPTER=kf` 但漏配 `WECOM_CORP_ID`/`WECOM_KF_SECRET` → 进程正常启动、回调正常监听,但取 token/拉消息永远静默失败。
- **验收**:上述三种错配均在启动 3 秒内给出中文指引并退出。

## P1 — 显著提升采用

### 2. 微信客服真机公网回调联调(Owner: 双方共同,S)
- 代码已完整(加解密 FIPS-197 钉住、17 项离线测试),但与腾讯服务器的端到端互通从未验证(README/docs 多处标注"待验证")。这是"最后一公里",不做 M1 无法闭合。
- **验收**:企微后台配置回调后,真实用户消息 5 秒内收到 AI 回复;处理失败回 500 腾讯重试不丢消息。
- **依赖**:需公网 HTTPS(用户侧资源)——可先文档化联调 SOP(内网穿透/云服务器两条路径),等用户醒来提供企微后台权限。

### 3. 转人工结构化原因分类与准召评测(Owner: CONV-C 算法,M)
- 现状:`mark_needs_human("agent 判定需人工介入")` 永远同一字符串;运营者不知道是退款/知识未命中/砍价,无从决定补知识还是加人手;评测集对"应否转人工"零自动化断言。
- **验收**:`escalation_reason` 透出 ≥3 类可区分标签;evals 对转人工触发有自动断言(漏召=客诉,误召=浪费人力,两个方向都测)。

### 4. 运营后台产品化验收(Owner: CONV-B,余量 S)
- `admin_console.py` 已交付并过三镜头审查(#5/#7,断线提示/失败反馈已修)。剩:移动端适配自查、README 级运营者上手指引。
- **验收**:不懂代码的运营者 5 分钟内独立完成"登录→收件箱→接管→人工回复→交回",全程零 API 文档。

### 5. 部署路径文档化 + 一键健康检查(Owner: CONV-B 文档 / CONV-C 脚本,S)
- Docker/compose/Makefile 已交付(`07f4b8c`/`e339184`,CI docker job 绿)。剩:README「生产部署」极简章节(填 .env → `docker compose up -d` → 验证)、`.env` 逐项填写指南、一键健康检查(三服务各自 ready 的一条命令)。
- **验收**:非技术运营者按文档 15 分钟从 `git clone` 到第一条真实回复。

## P2 — 远期(规模化再投)

### 6. 合规红线换版回归门禁(Owner: CONV-C,M)
在线 21/21 已含 4 个合规案例,但依赖真 key 属手动;目标:换模型/改 persona 必跑的合规专项(可 LLM-as-judge),不过全绿不合并。

### 7. 商品库向真实 SKU 规模平滑迁移(Owner: CONV-C,L)
`_is_hit` 子串匹配对规格变体("250g"↔"半斤")召回≈0;JSON 无 schema 校验(语法错误→静默空库)。做轻量 schema 校验 + 同义词/规格标准化 + ERP 导出迁移脚本模板。50-200 SKU 店铺可无痛迁移。

### 8. 可观测性骨架(Owner: CONV-C,M)

### 9. ntwork 企微群 hook(远期,待 Windows 实机)
合规灰色,保持 stub;文档已诚实披露风险与替代拓扑。
