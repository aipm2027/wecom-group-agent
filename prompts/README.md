# prompts/ 内容文件修改指南

> 这个目录是**店铺运营内容区**——最常被修改的地方。三个文件各管一摊,改之前花 2 分钟读完本文。
> ⚠️ 本 README 不会进入 AI 的提示词;但 `persona.md` 和 `knowledge.md` 的**每一个字**都会进,
> 所以那两个文件里**不要写维护注释/TODO**,维护说明只写在这里。

## persona.md — 客服的"性格与规矩"(改话术风格/红线/转人工时机)

- **结构**:角色定位 → 沟通风格 → 售后谨慎 → 安全防注入 → 合规红线 → 转人工触发 → 输出格式。改哪节就只动哪节,别整篇重写。
- **绝对不能删的机制**:转人工控制标记 `[[转人工]]` 的说明(代码靠它工作,见 `core/llm_handler.py` 的 `ESCALATE_TAG`);合规红线七条(有 `make gate` 门禁盯着);防提示注入节。
- **已知的坑(改转人工条款前必读)**:
  - "政策咨询"和"实际诉求"必须分开——"拆封了还能退吗"是咨询(知识库作答),"我要退款"才转人工。历史上混为一谈导致误召,详见 `docs/技术文档/09` A2/A3。
  - 砍价是阶梯:先用知识库标准话术回应,客户坚持才转人工。
- **改完必须跑**:`make gate`(需要 `.env` 有真实 `LLM_API_KEY`),0 失败 0 跳过才算改好。改了触发时机,还要在 `evals/golden.json` 补案例。

## knowledge.md — 店铺知识(改商品简介/活动/物流/售后 FAQ)

- **格式就是功能**:`# 一级标题` 分章节、`- ` 开头的列表项是一条条知识。RAG 检索按"章节标题+列表项"切块,乱改格式=检索变差。
- **各章节怎么用**:店铺(品牌/客服时间)/当前主推 & 活动(含"活动/主推"等词的条目会被 RAG 常驻召回,放最想让 AI 主动提的内容)/产品 & 价格(与 products.json 保持一致!两边价格不一致 AI 会精神分裂)/物流/售后退换/常见问答。
- **重要**:客户高频问的问题**必须在这里有答案**,否则 AI 会按"知识答不了→转人工"把咨询转给人工,白白消耗人力(真实案例:"退款几天到账"没写,就被转人工,见 `docs/测试文档/06` 记录 #5)。
- **改完必须跑**:`make eval`(离线,秒级);动了价格/政策再 `make eval-online` 抽查。

## products.json — 结构化商品库(加/改/删 SKU,精确查价)

- **字段规则**(schema 有校验,写错会在启动和 `make eval` 时报第几条错在哪):

| 字段 | 必填 | 规则 |
|---|---|---|
| `product_id` | ✅ | 字符串,唯一(重复会告警) |
| `name` | ✅ | 商品全名,客户消息里含它即命中 |
| `price` | ✅ | 数字(不要带引号、不要写"99元") |
| `category` / `spec` / `stock` / `promotion` | 可选 | spec 里写规格(如 `250g/袋`,克数能被"半斤"这类口语命中) |
| `keywords` | 强烈建议 | **字符串数组**,写客户口语的各种叫法(俗称/简称/错别字),召回全靠它 |
| `status` | 可选 | 只能是 `on_sale` / `off_sale` / `out_of_stock`,别自创 |

- **批量导入**:别手写几十条 JSON,用 `python3 examples/products_from_csv.py 你的导出.csv`(预览校验)→ 确认后加 `-o prompts/products.json` 写入。支持中文表头(商品编码/名称/售价…)。
- **与 knowledge.md 的关系**:products.json 管"精确查价/规格/库存",knowledge.md 管"话术级的产品介绍与活动"。价格两边都出现时**必须一致**。
- **改完必须跑**:`make eval`。

## 通用检查清单(内容改动收尾)

```bash
make check        # 编译+测试+离线评测
make eval-online  # 有 key 就跑:真模型抽查改动相关的问题
make gate         # 改了 persona 才需要;全绿才能上线
```
